# AI 与梦想系统完善 PRD

> 日期：2026-06-23
> 关联文档：`持续更新_05人物AI_Utility.md`、`持续更新_01性格.md`、`持续更新_03任务.md`、`持续更新_09人生路径与故事节点.md`、`持续更新_12行为埋点与调试.md`。
> 主要代码入口：`js/ai.js`、`js/systems/ai-candidate-provider.js`、`js/systems/ai-drama.js`、`js/systems/behavior-telemetry.js`、`js/config.js`、`js/admin.js`。

## 1. 背景

当前 AI 与梦想系统已经有雏形，但两者还没有形成闭环：

- AI 已有 Utility 评分、候选来源、作息锚点、社交冷却和调试日志，但仍偏“就近满足需求”。它能解释某个候选为什么分高，却不能稳定回答“这个人今天为什么把注意力放在这件事上”。
- 梦想系统已经有后台配置、人物分配、故事钩子和玩法字段，但首版只展示，不影响运行时行为、任务生成、叙事气泡或结局判断。
- 现有 `AiDrama.intent` 只是短时情绪/社交意图，不足以承载“长期梦想 -> 中期动机 -> 当前行动”的链路。
- 任务、人生路径、梦想、日程、性格都可能给 AI 加权，若没有分层规则，后续会变成不可调的权重汤。

本 PRD 的目标是把二者统一成一套三层动机结构：

```text
梦想 Dream：长期人生方向，回答“这个人这辈子想往哪里走”
Intent：当前动机，回答“这一段时间这个人把注意力投向什么”
Utility：动作选择，回答“在当前候选里哪件事最合适”
```

## 2. 设计原则

1. 不写死单个人物行为。宝玉、黛玉、凤姐只能作为验证样例，不能在 AI 底层硬编码。
2. 梦想不直接下动作。梦想只产生长期压力、偏好、阶段目标和故事钩子；具体动作仍由任务、Provider 和 Utility 选择。
3. Intent 不等于任务。Intent 是注意力方向，可被任务、需求、梦想、关系、身份、状态共同推高。
4. Utility 继续做底盘。它不理解人生意义，只对候选动作进行统一评分、冷却、去重、调度。
5. 调试优先。每一次关键 AI 选择必须能回放：梦想压力、当前 Intent、候选来源、评分因子、最终动作。

## 3. 当前问题

| 模块 | 当前状态 | 不完善点 |
|---|---|---|
| Utility AI | 已实现候选池、评分因子、作息锚点、社交冷却 | 缺少上层动机仲裁，候选仍容易被距离、当前需求和局部权重支配 |
| AI 日程 | 有 `scheduleWindows` 与 `routineAnchors` | 只管一天内节奏，不知道人物长期想做什么 |
| AiDrama Intent | 有短时社交/情绪意图 | 粒度偏窄，未覆盖梦想、职责、人生路径、长期目标 |
| 梦想配置 | 有 `dreamMetadata` 与 `dreamProfiles`，后台可编辑 | 不产生运行时压力、进度、事件、任务或结局结果 |
| 行为埋点 | 已记录候选、评分、作息 | 未记录梦想压力、Intent 候选、梦想进度变化 |
| 后台 v2 | AI 和梦想都有配置页 | 缺少联动视图：看不出某人物梦想如何影响 AI |

## 4. 核心概念

### 4.1 Dream 梦想

梦想是人物长期人生方向。它不替代性格，也不替代人生路径：

| 概念 | 回答的问题 | 例子 |
|---|---|---|
| 性格 | 这个人怎么反应 | 敏感、热络、爱洁、守时 |
| 梦想 | 这个人长期想把人生带到哪里 | 家门安稳、情缘圆满、治家掌事、翻案洗冤 |
| 人生路径 | 这个人处在哪条成长/身份线 | 丫鬟晋升、掌家、仕途、寄居客 |
| Intent | 这一段时间注意力投向什么 | 想读书、想探听、想照顾某人、想避风头 |
| Utility | 当前具体做哪件事 | 用书案、去找某人、接任务、回屋休息 |

### 4.2 Dream Pressure 梦想压力

梦想压力是 0 到 1 的运行时数值，表示该梦想当前有多需要被关注。

压力来源：

- 进度落后：长时间没有做与梦想相关的事情。
- 风险临近：梦想相关风险正在升高，例如家族钱少、关系恶化、病情变重、秘密暴露。
- 机会出现：相关人物、场景、任务、物品、事件在附近或可达。
- 情绪牵引：人物状态让某类梦想更急，例如受辱后更想翻案，寂寞时更想情缘圆满。
- 身份职责：主子、仆从、长辈、晚辈对梦想的可行路径不同。

压力不是成功率，也不是强制执行概率。它只决定梦想相关 Intent 更容易进入候选。

### 4.3 Intent 当前动机

Intent 是短中期动机，持续一段游戏时间，可被高优先级需求、任务、危机打断。

推荐第一版 Intent 类型：

| Intent ID | 来源 | 语义 | 典型候选 |
|---|---|---|---|
| `need:recover` | 基础需求 | 需求告急或明显偏低 | 吃饭、睡觉、梳洗、休息 |
| `routine:anchor` | 日程作息 | 当前时段该做某类生活行为 | 三餐、晨起梳洗、晚间社交 |
| `task:execute` | 任务系统 | 有已接任务或职责任务 | 任务步骤、找任务人物、去任务场景 |
| `social:bond` | 关系系统 | 想靠近、安慰、解释、争执 | 互动、找人、陪伴、道歉 |
| `dream:advance` | 梦想系统 | 推进长期梦想 | 读书、查账、探听、守护、游园 |
| `dream:protect` | 梦想系统 | 防止梦想受损 | 化解冲突、照顾病人、守密、避嫌 |
| `path:advance` | 人生路径 | 当前路径阶段目标 | 路径任务、关键故事节点 |
| `emotion:react` | 状态/戏剧 | 情绪驱动反应 | 哭、躲、发火、找人安慰 |

## 5. 运行时架构

```text
每次 AI 慢通道 tick
  1. 收集动机压力
     - Needs
     - Routine
     - Quest
     - Dream
     - Relation
     - LifePath
     - State/Drama
  2. 生成 Intent candidates
  3. 选择或保持当前 Intent
     - 高危需求/任务可抢占
     - 普通 Intent 有稳定时间，避免频繁跳变
  4. 把 context.intent 传给 Provider
  5. Provider 生成候选动作
  6. Utility 统一评分并执行
  7. 记录 telemetry
```

### 5.1 分层职责

| 层 | 能做 | 不能做 |
|---|---|---|
| DreamSystem | 计算梦想压力、进度、风险、可用钩子 | 不能直接塞动作队列 |
| IntentSystem | 选择当前动机，给 Provider/Utility 提供上下文 | 不能跳过任务/礼法/场景权限 |
| Provider | 根据 Intent 提供更合适的候选 | 不能自己决定最终动作 |
| Utility AI | 评分、去重、冷却、执行 | 不负责判断人生目标是否完成 |
| Telemetry | 记录可解释链路 | 不影响权重 |

## 6. 梦想数据协议

现有字段保留，并新增运行时可读字段。

```js
dreamMetadata: {
  family_stability: {
    label: '家门安稳',
    category: '家业',
    description: '盼家族不散、财用不竭、灾祸不临门。',
    examples: ['守住公账', '避免大祸', '照看核心亲眷'],
    storyHooks: ['账册缺口', '族中急病', '外头风声'],
    failureHooks: ['公账亏空', '亲眷离散', '抄检逼近'],

    conditions: {
      moneyMin: 200,
      disasterFreeDays: 7
    },

    gameplay: {
      intentTags: ['family', 'money', 'protect'],
      providerHints: ['quest', 'social', 'homeward'],
      actionWeights: { task: 1.2, comfort: 1.1 },
      riskSignals: [
        { type: 'familyFundBelow', value: 160, pressure: 0.4 },
        { type: 'coreMemberMoodBelow', value: 35, pressure: 0.25 }
      ],
      opportunitySignals: [
        { type: 'questCategory', value: '家务', pressure: 0.25 },
        { type: 'nearFamilyMember', pressure: 0.15 }
      ]
    }
  }
}
```

人物分配：

```js
dreamProfiles: {
  daiyu: {
    type: 'family_stability',
    title: '癸酉本式补天',
    description: '才情不只为自伤，也能替风雨里的贾府留一线生机。',
    condition: '把贾府金钱稳定在 200 以上，连续 7 日不触发大祸。',
    progress: {
      score: 0.35,
      lastTouchedAt: 12340,
      milestones: { moneyStable: true }
    }
  }
}
```

说明：

- `conditions` 是结局/成就口径。
- `gameplay` 是运行时压力和候选偏好。
- `progress` 可以先存在运行时存档，不要求写入默认配置。

### 6.1 梦想达成条件能力总表

### 6.2 第一批必须补的底层能力

P0 详细拆分见 `持续更新_19_p0_AI梦想条件底座.md`；P1 实体系统拆分见 `持续更新_19_p1_AI梦想实体系统.md`。本文只保留总览。

| 优先级 | 能力 | 为什么必须补 | 最小数据结构 |
|---|---|---|---|
| P0 | `DreamConditionRegistry` | 统一判定不同梦想条件，避免每个梦想硬写逻辑 | `conditionKey -> evaluate(ctx, spec)` |
| P0 | `DreamProgressStore` | 记录条件计数、最近推进、失败原因 | `{ charId, dreamType, counters, milestones, lastTouchedAt }` |
| P0 | `BehaviorDailyStats` | 支持“连续 N 日”“每日最低值”“独处日”“娱乐日” | `{ charId, day, needMins, actionTagMinutes, sceneIds }` |
| P0 | `HealthSystem` | 梦想大量依赖健康、病中、重病、照顾病弱 | `health 0..100, illnessState, sickDays, careEvents` |
| P1 | `SocialIssueSystem` | 误会、流言、旧怨、冲突、翻案都应是同一种可解决 issue | `{ id, type, targetIds, severity, resolvedBy, resolvedAt }` |
| P1 | `SecretSystem` | 秘闻、泄露、翻案、证据都需要秘密账本 | `{ id, ownerId, targetIds, severity, knownBy, exposedAt, clearedAt }` |

## 7. DreamSystem 第一版能力

### 7.1 API

```js
DreamSystem.getProfile(charId)
DreamSystem.getMetadata(dreamType)
DreamSystem.evaluate(char)
// returns:
{
  type,
  label,
  pressure,       // 0..1
  progressScore,  // 0..1
  riskScore,      // 0..1
  opportunityScore,
  intentTags,
  providerHints,
  reasons: []
}
```

### 7.2 压力计算

第一版使用可解释的加权模型：

```text
pressure =
  progressLag * 0.30
  + riskScore * 0.35
  + opportunityScore * 0.20
  + personalityFit * 0.10
  + identityFit * 0.05
```

约束：

- `need:recover` 高危需求优先级永远高于普通梦想。
- 已接近截止的任务优先级高于普通 `dream:advance`。
- `dream:protect` 可以高于普通作息，但不能越过礼法、场景权限、行动能力限制。

## 8. IntentSystem 第一版能力

### 8.1 Intent candidate

```js
{
  id: 'dream:advance:family_stability',
  type: 'dream:advance',
  label: '推进梦想：家门安稳',
  pressure: 0.68,
  priority: 40,
  tags: ['dream', 'family', 'money', 'protect'],
  providerHints: ['quest', 'social'],
  reason: '公账偏低，且附近有可下发家务任务',
  source: { system: 'dream', dreamType: 'family_stability' },
  durationGameMin: 90
}
```

### 8.2 选择规则

1. 生成所有候选 Intent。
2. 按 `priority * pressure * stability` 排序。
3. 若当前 Intent 仍有效且新候选分数没有超过 `current * 1.25`，保持当前 Intent。
4. 高危需求、任务截止、剧情强制节点可以抢占。
5. Intent 完成、过期、目标不可达时释放。

### 8.3 与 Utility 的关系

Provider 可以读取：

```js
context.intent = {
  type,
  tags,
  providerHints,
  targetCharId,
  dreamType
}
```

Utility 评分新增：

```text
intentFactor =
  matchingTags ? 1.15..2.0
  unrelatedButHarmless ? 0.85..1.0
  conflictsWithIntent ? 0.25..0.75
```

## 9. AI 行为体验目标

### 9.1 日常行为

- NPC 仍能自然吃饭、睡觉、梳洗、社交。
- 梦想不会让人物饿着肚子写诗、困到崩溃还查账。
- 一天内有生活节奏，几天内能看出人生方向。

### 9.2 梦想驱动差异

同样是“意趣低”：

- 诗书才名的人更容易去书案、联句、论道。
- 风月逍遥的人更容易游园、饮宴、找热闹。
- 清净自守的人更容易独处、读书、回屋。

同样是“关系紧张”：

- 护佑成全的人更想调停或照顾。
- 翻案洗冤的人更想找证据、找证人。
- 秘闻在握的人更想观察、探听、交换消息。

## 10. 任务与叙事联动

### 10.1 梦想任务

任务模板可增加：

```js
dreamTags: ['family', 'protect']
dreamTypes: ['family_stability']
dreamProgress: { key: 'moneyStable', delta: 0.1 }
```

作用：

- AI 有 `dream:advance` Intent 时，相关任务候选加权。
- 完成任务后推进梦想进度。
- 失败任务可增加梦想风险或触发 failureHook。

### 10.2 叙事气泡

新增气泡触发：

| Trigger | 说明 |
|---|---|
| `dream_pressure_high` | 梦想压力超过阈值 |
| `dream_progress_up` | 梦想进度提升 |
| `dream_risk_high` | 梦想风险升高 |
| `dream_blocked` | 梦想相关 Intent 连续失败 |
| `dream_milestone` | 梦想阶段节点达成 |

### 10.3 人生路径

人生路径可以读取梦想，但不被梦想替代：

- 人生路径定义身份和阶段线。
- 梦想定义个人长期愿望。
- 同一路径下不同梦想产生不同玩法重点。

例：同为丫鬟晋升路径，`护佑成全` 偏守护主子，`秘闻在握` 偏消息与风险，`安身养命` 偏稳定归属。

## 11. 后台 v2 需求

## 13. 验收标准

### P0 验收

- 后台能看到当前人物梦想压力、当前 Intent、Utility Top 候选。
- 连续 3 日模拟中，NPC 不因梦想忽略基础生存需求。
- 至少 5 类梦想能影响 Intent 候选，而不是只影响展示文本。
- `BehaviorTelemetry` 可回放一次选择的完整链路：梦想压力 -> Intent -> Provider -> Utility。

### P1 验收

- 完成梦想相关任务能推进梦想进度。
- 梦想风险升高时能触发叙事气泡或事件钩子。
- 同一性格不同梦想的人，在 3 日行为统计中能看出差异。
- 同一梦想不同身份的人，推进方式不同。例如主子通过下发/理事，仆从通过执行/传话/守密。

### P2 验收

- 梦想可进入结局统计。
- 人生路径阶段目标可与梦想互相引用。
- 后台支持梦想信号的结构化编辑，而不是只改 JSON。
- 长模拟报告输出每个角色梦想推进率、失败原因、最常见阻断。

## 14. 分阶段实施计划

### Phase 0：PRD 与边界统一

- 明确 Dream / Intent / Utility 三层边界。
- 梦想仍保留“长期驱动”，不恢复“追求”概念。
- 后台 v2 保留 AI 和梦想配置页。

### Phase 1：只读 DreamSystem + Intent 候选

- 新增 `DreamSystem.evaluate(char)`，只读现有配置和运行时。
- 新增 `IntentSystem` 或在 AI 内部先实现 `collectIntentCandidates`。
- `ai:decision` 记录当前 dream/intent 信息。
- 不改变任务、不改变结局。

### Phase 2：Provider 与 Utility 接入

- Provider 读取 `context.intent`。
- Utility 增加 `intentFactor`。
- 梦想相关候选可被加权，但基础需求和任务截止仍可抢占。
- 后台显示 Dream -> Intent -> Utility 链路。

### Phase 3：梦想进度与叙事

- 完成任务、互动、事件时更新梦想进度。
- 高压力/高风险触发叙事气泡。
- failureHooks 可进入搞事情/事件系统。

### Phase 4：结局与长模拟

- 长模拟报告输出梦想推进率。
- 结局统计读取梦想进度。
- 人生路径与梦想组合生成不同结局提示。

## 15. 不做范围

- 不做 LLM 直接控制 AI 行为。
- 不做单个人物硬编码行为。
- 不把梦想变成强制任务队列。
- 不让梦想绕过礼法、场景权限、行动能力、基础需求危机。
- 不在第一版做完整 HTN Planner，任务分解仍沿用现有任务系统。

## 16. 关键结论

AI 不完善的根因不是“权重还不够多”，而是缺少上层动机。梦想不完善的根因不是“梦想类型还不够多”，而是缺少运行时压力、进度和反馈。

下一步应优先实现：

1. `DreamSystem.evaluate`
2. Intent 候选与选择器
3. `context.intent` 传入 Provider
4. `intentFactor` 进入 Utility
5. 后台和 telemetry 展示完整链路

这样 AI 会保留自由模拟的弹性，梦想也会从“人物设定文本”变成能驱动日常、任务、叙事和结局的长期人生方向。
