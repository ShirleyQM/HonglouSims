# 社交互动与对话 PRD

> 更新时间：2026-07-09
> 当前口径：一级大类固定六类并普遍展示；AI 和 UI 共用同一套互动意愿公式；关系/轴/礼法多数走软锁，不再硬隐藏；菜单外观不再用意愿强弱做差异。礼法不是硬锁，而是进入“互动意愿”和“逾矩风险”的连续变量。
> 主要代码：`js/action-queue.js`、`js/ui.js`、`js/systems/interaction-social.js`、`js/systems/interaction-score.js`、`js/systems/identity-protocol.js`、`js/systems/interaction-depth.js`、`js/systems/multi-interact.js`、`js/systems/interaction-llm.js`、`js/ai.js`。

## 1. 总目标

社交互动系统负责把“点一个人”变成可解释、有礼法、有情绪后果的互动链：

- 一级大类让玩家理解“此刻可以往哪个方向互动”。
- 二级模板让玩家选择具体行为，如寒暄、安慰、打趣、私语、讥讽。
- 关系、人物状态、身份礼法、场景见证者、失败热度共同影响推荐强度。
- 不合适或逾矩的方向不隐藏，而是以虚线/风险提示展示，让玩家感知规则。
- 失败不是单次空结果，会进入尴尬、心碎、愠怒、回怼、争执等深度链。

### 1.1 未来方向：规则引擎先行，接口对齐 Generative Agent

最终目标是把 NPC 社交推进到 Generative Agent 方向：角色拥有记忆、反思、关系牵挂、日程计划和可解释的下一步意图。但当前阶段不直接让 LLM 参与每次底层行动选择，而是先用规则引擎实现同一套输入输出协议。

核心原则：

- 输入一致：规则引擎与未来 LLM 引擎读取同一份 `SocialDecisionInput`。
- 输出一致：两者都产出同一份 `SocialDecisionResult`，下游 UI、Utility AI、行动队列、埋点和调试面板不关心中间由谁推理。
- 可解释性一致：规则引擎写结构化解释，LLM 未来也必须返回同结构解释，不能只给自然语言理由。
- 玩家可控：参考 Sims 系自主 AI 的边界，NPC 不做不可见的长程复杂规划；复杂计划必须能在 UI、气泡、日志或任务中被玩家理解。
- 当前实现：V1 使用规则、权重和阈值；LLM 只负责对白、尴尬收场文案、日终总结或离线反思，不直接控制动作。

调研结论简表：

| 来源 | 可借鉴点 | 本项目取法 |
|---|---|---|
| Sims 式 autonomy | 需求、性格、物体/互动 affordance 共同产生下一步动作；不强调长程规划。 | 保留 Utility AI 作为日常底盘，避免 NPC 过度自演。 |
| Sims 开发经验 | 复杂自主计划必须被玩家看懂，否则像 bug；过强自主会削弱玩家讲故事。 | 所有社交意图必须进调试、气泡或日志；玩家手动操作优先。 |
| Generative Agents | 记忆、反思、检索和计划可以生成 believable social behavior。 | 先设计同构输入输出，未来把规则推理器替换为 LLM/混合推理器。 |
| Social Practice 模型 | 社交不是直接选动作，而是先选“情境实践”：问候、宽慰、避嫌、试探、管束等。 | 新增“目标牵挂 + 社交实践”层，专门处理为什么找某人、以什么分寸接近。 |

统一输入建议：

```js
SocialDecisionInput = {
  actorId,
  candidateTargetIds,
  time: { day, hour, period },
  location: { sceneId, sceneType, crowd, witnesses },
  actorSnapshot: {
    needs, activeStates, traits, specialties, dreams,
    role, socialRank, currentTask, recentActions,
  },
  targetSnapshots: [{
    targetId,
    relationAxes: { affection, trust, friendship, submissionAtoB, submissionBtoA },
    relationLabels,
    recentMemories,
    lastInteractionAt,
    currentState,
    accessibility,
  }],
  worldContext: {
    order, economy, etiquetteRules, familyRoles, activeEvents,
  },
};
```

统一输出建议：

```js
SocialDecisionResult = {
  targetDesires: [{
    targetId,
    score,
    desireType: 'miss' | 'duty' | 'comfort' | 'court' | 'avoid' | 'confront' | 'opportunistic',
    reasons: [{ key, label, value, weight }],
  }],
  practiceCandidates: [{
    targetId,
    practiceId,
    category,
    score,
    risk,
    reasons,
  }],
  interactionCandidates: [{
    targetId,
    interactionId,
    willingness,
    autonomyAllowed,
    expectedOutcome,
    reasons,
  }],
  selected: {
    targetId,
    practiceId,
    interactionId,
    provider: 'rule' | 'llm' | 'hybrid',
    confidence,
    explanation,
  },
};
```

规则引擎和未来 LLM 引擎都必须遵守该输出结构。LLM 可以生成更细腻的 `explanation` 和 `reasons`，但不得绕过 `autonomyAllowed`、礼法风险、行动队列、任务和硬条件校验。

## 2. 一级互动大类

一级大类固定为六类，前台全部展示，不因不合适而隐藏：

| 显示顺序基准 | 配置 ID | 展示名 | 语义边界 |
|---|---|---|---|
| 1 | `weijie` | 宽慰 | 倾听、安抚、陪伴、照拂；对方低落、尴尬、心碎或愠怒时更推荐。 |
| 2 | `xujiu` | 闲谈 | 寒暄、问安、闲聊、打听；最稳妥的日常往来。 |
| 3 | `lundao` | 论艺 | 论诗、评文、对弈、品茗、谈禅；承接技能、兴趣与雅集场景。 |
| 4 | `chuanqing` | 传情 | 示好、私语、亲近、暧昧；强关系驱动，常受礼法、场景和见证者影响。 |
| 5 | `tiaoxiao` | 玩笑 | 打趣、调侃、揭短、嬉闹；关系熟时轻快，关系低或身份不合时容易冒犯。 |
| 6 | `zhengchi` | 争执 | 顶嘴、斥责、讥讽、吵架；关系差、愠怒、反复失败时更靠前。 |

历史 ID 继续保留，避免重建模板、任务、AI 权重和多人互动引用。只改变玩家可见名称与礼法中文分类。

## 3. 一级菜单展示逻辑

一级大类“普现”，但根据当前上下文改变排序与视觉：

| 状态 | UI 表现 | 含义 |
|---|---|---|
| 推荐 | 排序靠前 | 状态、关系、任务、性格或失败链鼓励此方向。 |
| 普通 | 实线普通框 | 可以自然发生，但没有特别强动机。 |
| 风险 | 不额外改变外观，仅在 hover/title 或执行结果中体现 | 主导子项关系不足/逾矩，或大类本身存在明显礼法风险。 |
| 选中/hover | 黄色发光 | 用于玩家当前指向或选中的菜单项，不用于表达意愿强弱。 |
| 不太合适 | 弱化但仍展示 | 当前子项大多不可用或容易失败；点击后仍能看到原因。 |

推荐强度由以下信息综合：

```text
一级大类推荐分 =
  子互动最高意愿
+ 目标状态修正
+ 发起人状态修正
+ 双方关系修正
+ 身份礼法修正
+ 逾矩/低分风险修正
+ 技能/兴趣倾向修正
+ 失败热度与冲突状态修正
```

实现口径：

- `getAvailableInteractions()` 仍返回每个大类下的模板和 `checkInteractionAvailable()` 结果。
- `renderImCategories()` 对每个大类汇总子项意愿、低分、风险和礼法行为；意愿只参与排序和推荐，不再直接改变一级/二级外观。
- `IdentityProtocolSystem.evaluateProtocolBehavior()` 接入一级大类，用 `allowed / conditional / risky / forbidden` 影响虚线风险和排序。
- `InteractionScoreSystem.socialWillingness()` 的子项意愿用于大类推荐强度。
- 二级互动不显示“意愿高/中/低/冷”文字，也不再用发光/虚线表达意愿；所有可点二级项保持同一套基础样式，hover/点击/当前项才发光。
- 目标状态如 `ganshang / heartbroken / awkward / sullenAnger / angry / offended` 会提高“宽慰”或“争执”。
- 高关系、暗生情愫、正向心境会提高“传情”；但身份或场景风险仍以虚线提示。
- “玩笑”在关系熟时提高，目标伤心或愠怒时降低并更容易显示风险。
- 排序只在打开菜单时计算，避免菜单打开后跳动。

## 4. 二级互动模板

`interactionTemplates` 是具体互动主表。每条模板属于一个一级大类：

```text
interactionCategories 大类
  -> interactionTemplates 子互动
  -> category 字段指向大类 ID
```

模板字段继续沿用当前解释器：

| 字段 | 用途 |
|---|---|
| `id` / `name` / `category` / `type` | 模板标识、展示名、所属大类、互动形态。 |
| `relMin` / `relMax` / `minScore` | 关系意愿阈值，关系不足进入低分收场。 |
| `duration` / `cooldown` / `once` | 执行时长、冷却、一次性限制。 |
| `skillReq` / `requireFurniture` / `furnitureReq` | 技能或家具条件。 |
| `lines` | 非 LLM 回退对白。 |
| `lineVariants` | 可选文案模板池；按关系分数、礼法风险和语气池选择一条运行时对白，`lines` 仍作兜底。 |
| `effects` | 需求、状态、记忆、旧关系变化。 |
| `axisEffects` | 好感、信任、亲密、服从等关系轴变化。 |
| `onLowScore` | 低意愿失败后果和失败文案。 |
| `risky_details` | 逾矩成功率、失败状态、见证者惩罚、风险失败文案。 |
| `contact_type` / `is_risky` | 身体接触与逾矩风险入口。 |
| `llmPrompt` | 本地 LLM 话题/风格标签。 |

查重规则：新增互动前必须先比较同名、同大类、同接触类型、同效果和同玩法目的。只是补文案、失败状态、低分收场或 LLM 标签时，更新原模板，不创建重复模板。

## 5. 身份礼法接入

身份礼法同时影响一级大类和二级互动：

- 一级大类读取中文分类名，如 `闲谈 / 论艺 / 玩笑 / 宽慰 / 传情 / 争执`。
- `identityProtocolConfig.interactionCatMap` 将模板 ID 映射到礼法分类。
- `identityProtocolConfig.rules` 按 `rel + cat` 输出 `allowed / conditional / risky / forbidden`。
- `contactTypeRules` 按 `contact_type` 再叠加身体接触风险。
- 礼法风险不直接隐藏一级大类，而是虚线和提示；`forbidden` 不再从菜单硬拦，执行时进入逾矩/失败链，成功率可为 0。
- 传情等强语气互动可用 `lineVariants.good / poor / elegant / plain / blunt / subtle` 分池；文案不写固定称呼，若需要点名使用 `{A}` / `{B}` 占位，由运行时替换人物名，身份称谓仍交给身份礼法上下文提示。
- 逾矩失败文案可用 `risky_details.identity_fail_lines[身份关系].fail/repeat/witnessed` 覆盖泛用文案，主仆场景优先写装傻、沉默、赔笑、避重就轻，避免下人对主子使用平辈翻脸口吻。

### 5.1 礼法意愿公式

社交互动不要用关系或礼法做硬解锁。互动是否展示、玩家能否尝试、NPC 是否愿意主动做，是三件事：

| 层级 | 玩家 UI | NPC AI | 说明 |
|---|---|---|---|
| 展示 | 一级大类普遍展示，二级模板尽量展示 | 不影响候选生成 | 让玩家看见世界规则，而不是猜“为什么没有”。 |
| 意愿 | 用排序、提示和失败预期表达 | 进入候选权重 | 关系高低、性格、礼法、情绪共同影响“想不想做”。 |
| 结果 | 执行后可能成功、冷场、拒绝、逾矩失败、被旁观 | 结算同玩家行为 | 失败与逾矩进入状态、记忆、关系、秩序和声望。 |

推荐统一公式：

```text
互动意愿 =
  基础意愿
+ 关系吸引
+ 情绪/需求压力
+ 性格倾向
+ 场景机会
- 礼法压力
- 失败热度
- 冷却/疲劳
```

礼法压力拆成：

```text
礼法压力 =
  身份差压力
+ 名分/亲缘/主仆边界
+ 身体接触风险
+ 场景严肃度
+ 见证者压力
+ 惩罚/名声风险
+ 环境秩序压力
+ 人物守礼倾向
- 人物冲动/任性倾向
```

其中：

- `身份差压力`、`名分/亲缘/主仆边界`、`身体接触风险` 由 `IdentityProtocolSystem` 和 `interactionTemplates.contact_type / is_risky` 提供。
- `见证者压力` 由当前场景人数、旁观者身份、公开/私密场景计算。
- `环境秩序压力` 读取秩序系统中的法度、体面等环境参数。
- `人物守礼倾向`、`人物冲动/任性倾向` 读取性格系统 effects，不在互动模板里单独写死。
- 关系高低只影响该互动的意愿，不负责“解锁/隐藏”。低关系可以尝试，但更可能冷场、拒绝或冒犯。

### 5.2 NPC 自主互动边界

NPC 自主行为要比玩家点击更克制：

| 礼法结果 | 玩家点击 | NPC 自主 |
|---|---|---|
| `allowed` | 正常展示和执行 | 正常进入候选池。 |
| `conditional` | 正常展示，提示分寸/场合 | 降权进入候选池。 |
| `risky` | 展示，执行时进入逾矩成功率和失败链 | 大幅降权；只有高关系、强情绪、低见证、低守礼或特殊状态时才选。 |
| `forbidden` | 展示为高风险或禁忌尝试，成功率可为 0 | 默认不自主选择；只允许醉酒、崩溃、剧情/任务强制、特殊性格爆发等入口。 |

每个高风险互动都应尽量配置“合礼替代动作”。AI 优先选择替代动作，玩家仍可选择越礼动作。

示例：

| 逾矩动作 | 合礼替代 |
|---|---|
| 拥抱 | 递帕、低声宽慰、陪坐片刻 |
| 背人 | 搀扶、叫丫鬟来扶 |
| 调笑 | 含蓄打趣、试探一句 |
| 训斥 | 提点、冷脸、命人传话 |

### 5.3 性格接入礼法

礼法相关性格不直接替代 `IdentityProtocolSystem`。它们只改变“压力”和“冲动”的权重：

```text
知道规矩：规矩 skill `ritual`
愿不愿守：守礼/任性、端方/放诞、谨慎/冲动等性格
敢不敢破：情绪、需求、关系、场景机会、见证者
怕不怕后果：名声、身份、秩序、惩罚、第四轴关系
```

当前推荐接入口：

- `TraitEffectSystem.protocolPressureMultiplier(c, context)`：修正礼法压力。
- `TraitEffectSystem.protocolImpulseMultiplier(c, context)`：修正越礼冲动。
- `TraitEffectSystem.protocolWitnessMultiplier(c, context)`：修正见证者压力。
- `TraitEffectSystem.protocolShameMultiplier(c, context)`：修正失败后的羞耻、尴尬、名声伤害。

示例：

| 场景 | 一级表现 |
|---|---|
| 平辈关系好，对方心情轻快 | `玩笑` 可能高亮。 |
| 仆从对主子玩笑 | `玩笑` 虚线，提示失分寸风险。 |
| 对方心碎或尴尬 | `宽慰` 高亮。 |
| 公开场景传情 | `传情` 可高亮但虚线，提示旁人眼杂。 |
| 对方愠怒且关系紧张 | `争执` 排序靠前。 |

## 6. 判定与执行流程

```text
玩家点人物
  -> getAvailableInteractions 汇总六类模板
  -> InteractionSocialSystem.evaluate 检查能力/物理硬条件和风险
  -> IdentityProtocolSystem 计算一级/二级礼法行为
  -> InteractionScoreSystem 计算关系意愿、礼法压力和低分收场
  -> UI 对一级大类排序、高亮、虚线风险
  -> 玩家选择二级互动
  -> DialogueSystem 或 InteractionLlmSystem 生成对白
  -> ActionQueue 结算效果
  -> Relation / State / Need / Memory 更新
  -> InteractionDepth / MultiInteract / AiDrama 接收事件
```

## 7. 互动失败深度

失败不是单次骰子结果，而是人物关系的负反馈入口。

| 入口 | 记录内容 | 结果 |
|---|---|---|
| 低分收场 | `mode=low_score`，同一发起人、对象、互动大类累计失败次数和热度 | 文案从冷场升级到不悦、回怼、冲突。 |
| 逾矩失败 | `mode=risk_fail`，额外考虑见证者和公开场景 | 更容易加热度，并触发羞恼、冒犯、旁观事件。 |
| 人物状态 | 读取双方 `activeStates`，按 `statusHeatModifiers` 修正热度 | 愤怒、被冒犯、感伤、愠怒更容易升级；开心、满足、暗生情愫降温。 |
| 成功互动 | `recordSuccess` | 消减同组失败热度，避免一次失败永久记仇。 |

可见失败状态：

| 状态 ID | 展示名 | 用途 |
|---|---|---|
| `awkward` | 尴尬 | 普通冷场、公开失败、逾矩未成后的短时社交卡顿。 |
| `heartbroken` | 心碎 | 传情/亲密互动失败后，发起人的情感受挫。 |
| `sullenAnger` | 愠怒 | 被反复冒犯、被纠缠或公开难堪后，对象压着火气。 |

默认升级层级：`mild` 冷场 -> `annoyed` 不悦 -> `retort` 回怼 -> `public_heat` 被人瞧见 -> `severe` 冲突。

当前 V1 做“状态、关系轴、强制争执互动、事件广播”。打架、告状、处罚、声望流言后续由 `multiInteractConfig`、任务系统或声望系统消费 `interaction:failure_depth` / `interaction:failure_escalated` 事件。

## 8. 多人互动与 AI 接入

- `multiInteractConfig.observerReactions` 监听互动、状态、场景进入、需求危机等事件。
- 旁观者可发气泡、加状态、强制互动、下发任务、提升 AI 行动权重。
- `MultiInteractSystem.getActionBoost()` 可临时提高某类互动或行动标签权重。
- AI 社交候选仍从 `interactionTemplates` 生成，读取可用性、互动意愿、起居/Utility AI、性格和多人加权。
- AI 与 UI 共用 `InteractionScoreSystem.socialWillingness()`：UI 用 `strength` 控制排序、提示和失败预期，AI 用 `factor` 进入候选权重。
- 一级大类排序主要服务玩家 UI；二级子项意愿和 NPC 自主社交候选共用同一套底层公式。
- NPC 自主互动必须消费同一份礼法压力和性格冲动计算，不能另写一套“关系够就做”的快捷逻辑。

## 9. LLM 对话口径

LLM 是表现层，不改变底层规则：

- 开关和配置维护在 `narrativeBubble.settings.llm`。
- 默认 provider 为本地 Ollama，可通过 `serve.mjs` 代理。
- 模型生成失败时回退模板对白。
- 模型段落可以按轮次应用轻量关系效果，但不直接绕过互动模板主效果。
- `prompts/interaction-global.txt` 和 `prompts/interaction-topics.txt` 是文风主入口。

## 附：左下角标签来源现状记录（暂不细化）

`闲庭漫步`
来源：`statusText`
位置：`js/action-queue.js`、`world.js`、`js/ui.js`

逻辑：

- 角色初始化时默认 `statusText: '闲庭漫步'`。
- 角色没有行动、没有排队任务时，`processQueue/updateAction` 会把临时动作状态重置成 `闲庭漫步`。
- UI 左下角名字旁边的那个小标签显示的是 `c.statusText || '闲庭漫步'`。
- 所以它不是状态系统标签，不在 `activeStates` 里，只是“当前在干嘛”的兜底文案。

`话不投机`
来源比较多，但左下角看到的大概率也是 `statusText`。

主要逻辑：

- 互动低分失败时，`js/action-queue.js` 会执行：
  - 发起者：`statusText = 尴尬/尴尬收场/低分状态`
  - 目标：`statusText = '话不投机'`
- 互动失败结束时也会 log：`话不投机，尴尬收场`。
- 它还出现在关系配置里：`js/config.js`，比如友情轴低分档 `relationName: '话不投机'`。
- 状态定义里也有“话不投机”的描述，比如 `冷场`、`尴尬` 的 desc，但状态标签本身通常显示 `冷场` / `尴尬`，不是 `话不投机`。

真正的状态标签
来源：`c.activeStates`
位置：`js/systems/state.js`、`js/systems/state-defs.js`

逻辑：

- 调用 `applyState(c, stateId)` 后，会把 `{ id, remaining }` 放进 `c.activeStates`。
- 左下角状态行渲染 `c.activeStates`。
- 普通状态被添加时，`applyState` 还会顺手把 `c.statusText = 状态名`，所以状态名也可能临时出现在名字旁边。

结论：

- `闲庭漫步`、`话毕`、`与谁说话`、`行走中` 这一类，是 `statusText`，属于“当前动作/临时状态文案”。
- `尴尬`、`冷场`、`心碎`、`愠怒` 这一类，是 `activeStates`，属于真正的状态系统标签。
- 现在左下角名字旁边显示 `statusText`，下面状态行显示 `activeStates`。如果后续觉得 `闲庭漫步/话不投机/话毕` 不该像标签，可以把名字旁边那个改成“行动状态”，视觉弱化，真正状态只放下面一行。
