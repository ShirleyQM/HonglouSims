# 社交互动与对话 PRD

> 更新时间：2026-06-25
> 来源：`0609_02_对话引起状态变化.md`、`0609_03_对话轮次和群聊.md`、`0612_05_新增社交互动.md`、当前代码。
> 主要代码：`js/action-queue.js`、`js/systems/interaction-social.js`、`js/systems/interaction-score.js`、`js/systems/dialogue.js`、`js/systems/interaction-llm.js`、`js/systems/ai-drama.js`、`js/ai.js`、`js/ui.js`。

## 1. 总目标

社交互动系统负责把“点一个人”变成可解释、有风险、有后果的互动链：

- 互动能不能做：由对象、身份礼法、冷却、技能、场景和硬条件判断。
- 人物愿不愿意做：由关系四轴、状态、性格、心情和戏剧意图影响。
- 做完发生什么：关系轴、状态、需求、记忆、叙事气泡和后续 AI 意图更新。
- 文本如何表现：模板对白优先，可切换本地 LLM 对话。

## 3. 当前配置化边界

### 3.1 已经走配置的部分

这些内容目前主要由 `CONFIG` 驱动，策划可以逐步通过后台表格维护：

| 配置项 | 当前承接内容 | 说明 |
|---|---|---|
| `interactionTemplates` | 互动名、分类、关系阈值、时长、冷却、一次性、对白、效果、风险字段 | 社交内容主表。 |
| `interactionCategories` | 互动分类、菜单分组 | 决定环形菜单和后台分类。 |
| `interactionSocialConfig` | 部分社交评分、风险、失败处理参数 | 仍需继续 schema 化。 |
| `multiInteractConfig` | 旁观、多人反应、强制互动、发任务等规则 | 多人互动第一版配置入口。 |
| `identityProtocolConfig` | 身份礼法、接触类型、逾矩默认风险 | 负责身份/上下级/亲疏对互动的限制。 |
| `interactionDepthConfig` | 互动失败热度、分层文案、重复失败升级、主动回怼配置 | 失败不再只是重复冷场，而是进入关系恶化和冲突链。 |
| `dialogues` / `prompts` | 模板对白和 LLM 文风入口 | LLM 是表现层，不绕过底层互动规则。 |

### 3.2 仍明显写死在代码里的部分

这些内容目前仍由 JS 流程解释，不能只靠新增配置完成：

| 写死项 | 所在方向 | 说明 |
|---|---|---|
| 互动执行流程 | `ActionQueue / startInteraction` | 走近、站位、锁定双方、开始对白、结束、打断。 |
| 可用性判定顺序 | `checkInteractionAvailable` | 关系阈值、冷却、一次性、低分软失败、风险判定的组合顺序。 |
| 效果解释器 | `applyInteractionEffects` | 配置里的 `effect.type` 只有代码认识才会生效。 |
| AI 选互动逻辑 | `js/ai.js` | 谁找谁、何时主动社交、候选权重、距离/关系/起居权重。 |
| 多人互动执行器 | `MultiInteractSystem` | `reactionType` 如旁观、强制互动、下发任务的执行方式。 |
| 任务到互动的 fallback | `QuestSystem` | 诗/吟/联句倾向 204，读书倾向 203，陪伴倾向 403，否则回退 101。 |
| 非 LLM 对白回退 | `buildInteractionDialogue` | 模板不足时的对白拼接、轮次、说话人选择仍是代码逻辑。 |
| 风险/礼法公式 | `InteractionSocialSystem / InteractionScoreSystem` | 配置能给参数，但成功率、见证惩罚、失败状态结算仍由代码解释。 |
| 站位和镜头体验 | `walkToCharAdjacent / updateCamera` | 互动前靠近、互动锁、自动社交镜头保护属于运行时行为。 |

### 3.3 配置化迁移原则

目标不是先接数据库，而是先拆出稳定的“互动规则解释器”：

```text
配置表
  -> 条件解释器
  -> 效果解释器
  -> AI 权重解释器
  -> 对话/多人反应解释器
  -> 运行时执行器
```

- 短期继续使用 `JSON 配置 + localStorage/IndexedDB + 导入导出`。
- 中期补齐互动 schema：`conditions / effects / aiWeights / risk / dialogue / reactions / questHooks`。
- 长期需要账号、多人协作、云存档、配置版本发布时，再接后端数据库。
- SQLite 更适合本地桌面工具；网页试玩版优先考虑 `IndexedDB` 或 `Supabase/Postgres`。

## 4. 后台互动模板配置口径

后台【互动模板和多人互动】分成四块：

- 互动分类。
- 按分类管理的互动模板。
- 社交评分规则。
- 多人规则。

### 4.1 互动分类 `interactionCategories`

互动分类是环形菜单第一层，也是 AI 匹配互动语义的大类别。

后台只展示三列，不再要求策划编辑整段 JSON：

| 字段 | 含义 |
|---|---|
| `id` | 大类别 ID，例如 `xujiu`、`lundao`、`chuanqing`。 |
| `name` | 前台展示名，例如叙旧、论道、传情。 |
| `description` | 策划说明：这个大类适合什么场景、什么关系和什么 AI 倾向。 |

改分类 ID 时，需要同步：

- `interactionTemplates[*].category`
- `interactionSocialConfig.categoryMinScore`
- `interactionSocialConfig.categoryOnLowScore`

当前后台已做同步写回。

### 4.2 子互动模板 `interactionTemplates`

互动模板按大类别导航管理。进入某个大类别后，下方表格每一行就是一个子互动模板。

配置关系：

```text
interactionCategories 大类别
  -> interactionTemplates 子互动
  -> category 字段指回大类别 id
```

例子：

```text
叙旧 xujiu
  -> 101 寒暄
  -> 102 闲谈
  -> 103 问安
```

每个子互动的一级字段在后台直接展开：

| 字段 | 含义 | 后台展示 |
|---|---|---|
| `id` | 模板 ID | 只读展示。 |
| `name` | 子互动名称 | 文本输入。 |
| `category` | 所属大类别 | 由当前导航决定，不单独编辑。 |
| `type` | `dialogue` 或 `action` | 下拉。 |
| `relMin` / `relMax` | 关系自然区间 | 数字输入。 |
| `duration` | 执行时长 | 数字输入。 |
| `cooldown` | 冷却 | 数字输入。 |
| `once` | 是否一次性 | 勾选。 |
| `llmPrompt` | LLM 话题标签 | 文本输入。 |

复杂字段保留 JSON，但只编辑该字段本身，不再编辑整条模板 JSON：

| 字段 | 内容 |
|---|---|
| `skillReq` | 技能要求，例如 `{ "skill": "poetry", "level": 2 }`。 |
| `lines` | 模板对白数组，支持 `{A}`、`{B}`。 |
| `effects` | 需求、状态、关系、记忆等效果。 |
| `axisEffects` | 关系四轴变化。 |
| `onLowScore` | 意愿低时的拒绝、尴尬、关系下降等收场。 |
| `unlock_conditions` | 硬条件，例如状态、性别、年龄、家具、场景。 |
| `risky_details` | 逾矩风险、失败状态、见证者惩罚等。 |

### 4.3 社交评分规则 `interactionSocialConfig`

这不是互动内容，而是解释“能不能自然发生”和“低分怎么收场”的规则层。

| 字段 | 含义 |
|---|---|
| `witnessRadius` | 见证者半径。 |
| `showLockedInteractions` | 是否显示暂不可用互动。 |
| `categoryMinScore` | 各大类别最低自然意愿分。 |
| `categoryOnLowScore` | 大类别低分收场。 |
| `hierarchyOnLowScore` | 身份关系 + 大类别的特殊低分收场。 |
| `riskyDefaults` | 风险互动失败默认状态等兜底。 |

## 5. 判定流程

```text
发起互动
  -> InteractionSocialSystem.evaluate 硬条件
  -> IdentityProtocolSystem 计算身份礼法与接触风险
  -> InteractionScoreSystem 计算关系意愿和低分提示
  -> 菜单展示意愿强弱
  -> 执行互动
  -> DialogueSystem 或 InteractionLlmSystem 生成对白
  -> ActionQueue 结算效果
  -> Relation / State / Need / Memory 更新
  -> AiDrama / MultiInteract / Narrative 接收事件
```

## 6. 关系意愿口径

互动不再因关系值低而简单消失。当前口径：

- 世界规则仍然硬拦：无效对象、不能说话、硬禁忌、技能不足、冷却、场景条件。
- 关系值决定“自然程度”和“风险收场”，可显示为意愿高、中、低、冷。
- `onLowScore` 负责尴尬、难过、拒绝、关系下降等后果。

## 7. 互动失败深度

失败不是单次骰子结果，而是人物关系的负反馈入口。

当前已接入 `InteractionDepthSystem`：

| 入口 | 记录内容 | 结果 |
|---|---|---|
| 低分收场 | `mode=low_score`，同一发起人、对象、互动大类累计失败次数和热度 | 文案从冷场升级到不悦、回怼、冲突。 |
| 逾矩失败 | `mode=risk_fail`，额外考虑见证者和公开场景 | 更容易加热度，并触发羞恼、冒犯、旁观事件。 |
| 人物状态 | 读取双方 `activeStates`，按 `statusHeatModifiers` 修正热度 | 对象愤怒/被冒犯/感伤/愠怒时更容易升级，开心/满足/暗生情愫时降温。 |
| 成功互动 | `recordSuccess` | 消减同组失败热度，避免一次失败永久记仇。 |

默认分层：

| 层级 | 触发倾向 | 表现 |
|---|---|---|
| `mild` 冷场 | 首次失败 | 随机冷场/避开文案。 |
| `annoyed` 不悦 | 约第 2 次或热度升高 | 对象可能获得 `offended`，好感小幅下降。 |
| `retort` 回怼 | 约第 3 次或中高热度 | 对象可能获得 `angry`，好感/信任下降，并可能主动发起争执互动。 |
| `public_heat` 被人瞧见 | 失败被见证且热度高 | 双方更尴尬，关系惩罚更重，发出失败深度事件供旁观系统接入。 |
| `severe` 冲突 | 多次纠缠或高热度 | 对象更可能主动争执，后续可扩展到打架、告状、处罚。 |

失败深度已接入的可见状态：

| 状态 ID | 展示名 | 用途 |
|---|---|---|
| `awkward` | 尴尬 | 普通冷场、公开失败、逾矩未成后的短时社交卡顿。 |
| `heartbroken` | 心碎 | 传情/亲密互动失败后，发起人的情感受挫。 |
| `sullenAnger` | 愠怒 | 被反复冒犯、被纠缠或公开难堪后，对象压着火气。 |

配置口径：

```js
interactionDepthConfig: {
  enabled: true,
  decayGameMin: 240,
  heat: {
    low_score: 18,
    risk_fail: 28,
    witnessedBonus: 10,
    publicBonus: 8,
    successReduce: 0.45,
  },
  statusHeatModifiers: {
    target: {
      angry: 16,
      offended: 14,
      sullenAnger: 14,
      heartbroken: 8,
      ganshang: 8,
      joyful: -6,
      elated: -8,
    },
    initiator: {
      angry: 8,
      chikuang: 10,
    },
  },
  levels: [
    {
      id: 'retort',
      minCount: 3,
      minHeat: 54,
      states: [{ target: 'target', stateId: 'angry', prob: 0.45 }],
      axisEffects: [{ axis: 'affection', delta: -2 }],
      escalation: {
        type: 'force_interaction',
        templateIds: [601],
        probability: 0.45,
      },
      lines: { low_score: [], risk_fail: [] },
    },
  ],
}
```

模板级也可以覆盖失败文案：

```js
onLowScore: {
  lines: ['{B}只淡淡一笑，并不接话。'],
  repeatLines: ['{B}皱眉道：「怎么又说这个？」'],
}

risky_details: {
  fail_lines: [],
  repeat_fail_lines: [],
  witnessed_fail_lines: [],
}
```

当前 V1 只做“状态、关系轴、强制争执互动、事件广播”。打架、告状、处罚、声望流言作为下一层后果链，后续应由 `multiInteractConfig`、任务系统或声望系统消费 `interaction:failure_depth` / `interaction:failure_escalated` 事件。

## 8. LLM 对话口径

LLM 是表现层，不改变底层规则：

- 开关和配置维护在 `narrativeBubble.settings.llm`。
- 默认 provider 为本地 Ollama，可通过 `serve.mjs` 代理。
- 模型生成失败时回退模板对白。
- 模型段落可以按轮次应用轻量关系效果，但不直接绕过互动模板的主效果。
- `prompts/interaction-global.txt` 和 `prompts/interaction-topics.txt` 是文风主入口。

## 9. 后续待办

- 群聊的“分裂成小圈子”和“私语化”仍需正式状态。
- LLM 对话需要更清晰的安全字段：可输出情绪、关系倾向，但不能直接执行任意效果。
- 互动模板后台应增加批量查重、ID 分段、冷却合理性检查。
- 争执、安慰、拒绝、心动等结果链可沉淀为可复用 Outcome 表。
- 把 `effect.type`、`reactionType`、`condition.type` 统一登记成可扩展注册表，减少新增互动时改 JS。
- 失败深度事件后续应接入旁观者告状、流言声望、打架/拉架和任务处罚链。
