# AI 与梦想系统 P1：实体系统 PRD

> 日期：2026-06-23
> 父文档：`持续更新_19_AI与梦想系统联动.md`
> 前置文档：`持续更新_19_p0_AI梦想条件底座.md`
> P1 目标：把 P0 中只能“计数”的梦想条件，升级为可追踪、可解决、可流转的实体系统。

## 1. P1 范围

P1 只补五类实体能力：

| 能力 | 解决的问题 | 首版代码入口 |
|---|---|---|
| `SocialIssueSystem` | 误会、流言、旧怨、冲突、失礼、驱逐需要可创建、升级、解决 | `js/systems/social-issue.js` |
| `SecretSystem` | 秘闻、泄露、澄清、负向秘密、翻案需要账本 | `js/systems/secret.js` |
| `InventorySystem` | 作品、信物、礼物、证据、线索需要物品实体和流转 | `js/systems/inventory.js` |
| `AuthoritySystem` | 治家掌事、掌家权、任务执行力不能只靠身份位阶 | `js/systems/authority.js` |
| 分域声望 | 功名、家族、仆从、外部名声不能共用一个总分 | `js/systems/reputation-domain.js` |

P1 不做：

- 不做完整结局系统；离散、负向结局、仕途锁死放 P2。
- 不做复杂经济债务账本，只保留物品和声望对梦想的影响。
- 不做 LLM 生成秘密或案件；实体由任务、互动、后台和事件创建。
- 不做背包 UI 的完整游戏化，只做后台可查、可调试、可转移。

## 2. 与 P0 的关系

P0 让梦想条件“能判定”；P1 让条件背后“有东西”。

| P0 计数/条件 | P1 实体来源 |
|---|---|
| `misunderstandingMax` | `SocialIssueSystem` 中 `type=misunderstanding` 的未解决 issue |
| `rumorMax` | `SocialIssueSystem` 中 `type=rumor` 的未解决 issue |
| `conflictResolvedMin` | `SocialIssueSystem.resolve(issueId)` 的解决记录 |
| `secretLeakMax`、`harmfulLeaksMax` | `SecretSystem.leak(secretId)` |
| `discoveredSecretsMin` | `SecretSystem.discover(secretId, charId)` |
| `clearedNegativeSecretsMin` | `SecretSystem.clear(secretId)` |
| `craftedWorksMin`、`praisedWorksMin` | `InventorySystem` 中 `type=work/craft` 的物品事件 |
| `workAsClueMin`、`falseEvidenceMax` | `InventorySystem` 中 `role=clue/evidence` 的物品状态 |
| `authorityMin` | `AuthoritySystem.getAuthority(charId, familyId)` |
| `reputationMin` | 分域声望里的 `scholarly/official/family/servant/outside` |

P1 实现后，`DreamConditionRegistry` 应优先读取 P1 实体；若实体不存在，再回退到 P0 计数。

## 3. SocialIssueSystem

### 3.1 Issue 类型

| type | 中文 | 用途 | 常见梦想 |
|---|---|---|---|
| `misunderstanding` | 误会 | 情缘、亲密关系、错递信物 | `affection_fulfilled` |
| `rumor` | 流言 | 清净、归属、驱逐风险 | `quiet_reclusion`、`righteous_protection` |
| `grievance` | 旧怨 | 翻案、洗冤、积怨 | `grievance_redress` |
| `conflict` | 冲突 | 化解冲突、积德修福 | `karmic_merit` |
| `protocol_breach` | 失礼/逾矩 | 礼法风险、重大失礼 | `carefree_roaming` |
| `exile_risk` | 驱逐风险 | 稳定归属、离府风险 | `righteous_protection` |

### 3.2 数据结构

```js
{
  id: 'issue_001',
  type: 'rumor',
  title: '夜里私语被传开',
  targetIds: ['xiren'],
  sourceId: 'qingwen',
  familyId: 1,
  sceneId: 'yihongyuan',
  severity: 45,
  tags: ['rumor', 'protocol', 'servant'],
  status: 'open', // open/resolved/escalated/expired
  createdDay: 12,
  resolvedDay: null,
  resolvedBy: null,
  evidenceItemIds: [],
  relatedSecretIds: [],
  effects: {
    relationDelta: -5,
    reputationDelta: { family: -4 }
  }
}
```

### 3.3 API

```js
SocialIssueSystem.createIssue(payload)
SocialIssueSystem.getIssue(issueId)
SocialIssueSystem.findIssues(filter)
SocialIssueSystem.countOpen(filter)
SocialIssueSystem.resolveIssue(issueId, resolverId, method)
SocialIssueSystem.escalateIssue(issueId, delta, reason)
SocialIssueSystem.serialize()
SocialIssueSystem.apply(state)
```

### 3.4 梦想联动

- `resolveIssue(type=conflict)` 推进 `conflictResolved`。
- `resolveIssue(type=misunderstanding)` 降低 `misunderstandingMax` 风险。
- `createIssue(type=rumor, severity>=60)` 可推高 `dream:protect`。
- `protocol_breach` 写入失礼计数，用于刘姥姥“不触发重大失礼”。

## 4. SecretSystem

### 4.1 Secret 类型

| type | 中文 | 用途 |
|---|---|---|
| `personal` | 私事 | 人物隐私、把柄 |
| `financial` | 账目 | 公账亏空、挪用、债务 |
| `romance` | 情缘 | 信物、私情、误会 |
| `grievance` | 冤屈 | 翻案、洗白 |
| `protocol` | 礼法 | 逾矩、失礼、见证者 |

### 4.2 数据结构

```js
{
  id: 'secret_001',
  type: 'grievance',
  title: '晴雯被错疑偷物',
  ownerId: 'qingwen',
  targetIds: ['qingwen'],
  knownBy: ['xueyan'],
  severity: 70,
  harmful: true,
  status: 'hidden', // hidden/discovered/leaked/cleared
  evidenceItemIds: ['item_robe_patch'],
  createdDay: 8,
  discoveredDayByChar: { xueyan: 9 },
  leakedDay: null,
  clearedDay: null,
  tags: ['negative', 'grievance']
}
```

### 4.3 API

```js
SecretSystem.createSecret(payload)
SecretSystem.discover(secretId, charId, source)
SecretSystem.leak(secretId, actorId, targetIds, context)
SecretSystem.clearSecret(secretId, actorId, evidenceItemIds)
SecretSystem.findSecrets(filter)
SecretSystem.countKnown(charId, filter)
SecretSystem.serialize()
SecretSystem.apply(state)
```

### 4.4 梦想联动

- `discover` 推进 `discoveredSecrets`。
- `leak(harmful=true)` 增加 `harmfulLeaks`，并可创建 `rumor`。
- `clearSecret(tags includes negative)` 推进 `clearedNegativeSecrets`。
- 秘密可以引用物品证据，供 `grievance_redress` 使用。

## 5. InventorySystem

### 5.1 物品类型

| type | 中文 | 例子 | 梦想用途 |
|---|---|---|---|
| `work` | 作品 | 诗稿、文章、绣品 | 才名、巧艺 |
| `gift` | 礼物 | 扇套、络子、点心 | 情缘、关系 |
| `token` | 信物 | 旧帕、发簪、扇坠 | 情缘、误会 |
| `clue` | 线索 | 账页、旧信 | 翻案、秘闻 |
| `evidence` | 证据 | 修补痕迹、证词物件 | 翻案、洗冤 |
| `medicine` | 药物 | 药汤、药材 | 健康、护佑 |

### 5.2 数据结构

```js
{
  id: 'item_001',
  type: 'work',
  name: '海棠诗稿',
  ownerId: 'xiangyun',
  creatorId: 'xiangyun',
  holderId: 'baoyu',
  quality: 82,
  roles: ['gift', 'clue'],
  tags: ['poetry', 'praised'],
  relatedSecretIds: [],
  relatedIssueIds: [],
  createdDay: 10,
  transferLog: [
    { fromId: 'xiangyun', toId: 'baoyu', day: 10, reason: '赠诗' }
  ]
}
```

### 5.3 API

```js
InventorySystem.createItem(payload)
InventorySystem.getItem(itemId)
InventorySystem.findItems(filter)
InventorySystem.transferItem(itemId, fromId, toId, reason)
InventorySystem.addRole(itemId, role, reason)
InventorySystem.addTag(itemId, tag, reason)
InventorySystem.markPraised(itemId, actorId)
InventorySystem.serialize()
InventorySystem.apply(state)
```

### 5.4 梦想联动

- `createItem(type=work/craft)` 推进 `craftedWorks` 或 `works`。
- `markPraised` 推进 `praisedWorks` / `praise`。
- `addRole(role=clue/evidence/token)` 可推进 `workAsClue`。
- `transferItem` 可触发信物误会或礼物关系变化。

## 6. AuthoritySystem

### 6.1 掌家权来源

掌家权不是身份位阶本身，而是“能不能调动人、钱、规矩”的综合能力。

```text
authority =
  roleBase
  + rankBonus
  + managedTaskQuality
  + servantSubmission
  + familyReputation
  + recentIssuePenalty
```

| 来源 | 说明 |
|---|---|
| `roleBase` | 家主、配偶、长辈、掌事仆从有不同基础值 |
| `rankBonus` | 身份位阶只提供基础加成 |
| `managedTaskQuality` | 下发管理/家务任务的完成质量 |
| `servantSubmission` | 仆从对其服从轴平均值 |
| `familyReputation` | 家族/府内声望 |
| `recentIssuePenalty` | 近期流言、冲突、账目问题会扣分 |

### 6.2 API

```js
AuthoritySystem.getAuthority(charId, familyId)
AuthoritySystem.explainAuthority(charId, familyId)
AuthoritySystem.recordManagedTask(issuerId, assigneeId, category, quality)
AuthoritySystem.serialize()
AuthoritySystem.apply(state)
```

### 6.3 梦想联动

- `household_power.authorityMin` 读 `getAuthority`。
- `managedTasksMin` 读 `recordManagedTask` 计数。
- 掌家权低时推高 `dream:protect` 或 `dream:advance:household_power`。

## 7. 分域声望

### 7.1 声望域

| domain | 中文 | 典型来源 |
|---|---|---|
| `general` | 总体名声 | 默认兼容旧 `c.reputation` |
| `family` | 府内体面 | 家族事件、礼法、长辈认可 |
| `servant` | 下人间口碑 | 赏罚、照拂、苛责、仆从任务 |
| `scholarly` | 才名 | 诗社、论道、作品被赞 |
| `official` | 功名体面 | 课业、考试、差事 |
| `outside` | 外部名声 | 园外、人情往来、买办 |

### 7.2 数据结构

```js
char.reputationDomains = {
  general: 0,
  family: 0,
  servant: 0,
  scholarly: 0,
  official: 0,
  outside: 0
}
```

### 7.3 API

```js
ReputationDomainSystem.get(charId, domain)
ReputationDomainSystem.change(charId, domain, delta, reason)
ReputationDomainSystem.getBestDomain(charId)
ReputationDomainSystem.serializeChar(char)
ReputationDomainSystem.applyChar(char, state)
```

### 7.4 梦想联动

- `official_merit.reputationMin` 默认读 `official` 或 `scholarly`。
- `poetry_fame` 读 `scholarly`。
- `household_power` 读 `family` 和 `servant`。
- 仆从的“声望”主要读 `servant/family`，主子的“声望”主要读 `family/outside/official`。

## 8. 后台 v2 P1

梦想系统页在 P0 条件诊断下新增 P1 实体区：

| 区块 | 内容 |
|---|---|
| 社会问题 | 当前人物相关误会、流言、冲突、失礼、驱逐风险 |
| 秘密账本 | 已知秘密、被泄露秘密、已澄清负向秘密 |
| 物品/信物 | 作品、礼物、信物、线索、证据 |
| 掌家权解释 | 掌家权总分和来源拆解 |
| 分域声望 | 各域声望数值、最近变更来源 |
| 调试按钮 | 创建 issue/secret/item，推进、解决、泄露、转移 |

后台原则：

- 列表主入口必须结构化，不要求用户直接写 JSON。
- 高级 JSON 可以保留折叠区。
- 每个实体必须能跳回影响的梦想条件。

## 9. 事件与 Telemetry

新增事件：

| Event | 载荷 |
|---|---|
| `issue:created` | `issueId, type, targetIds, severity, tags` |
| `issue:resolved` | `issueId, resolverId, method` |
| `secret:created` | `secretId, type, ownerId, severity, harmful` |
| `secret:discovered` | `secretId, charId, source` |
| `secret:leaked` | `secretId, actorId, targetIds, harmful` |
| `secret:cleared` | `secretId, actorId, evidenceItemIds` |
| `item:created` | `itemId, type, ownerId, creatorId, roles` |
| `item:transferred` | `itemId, fromId, toId, reason` |
| `authority:evaluated` | `charId, familyId, authority, factors` |
| `reputation_domain:change` | `charId, domain, delta, value, reason` |

`DreamConditionRegistry` 的结果中要显示实体来源，例如“未解流言 2 条”“已知秘密 4 条”“掌家权 73”。

## 10. 验收标准

| 验收项 | 标准 |
|---|---|
| SocialIssue | 能创建、查询、解决误会/流言/冲突，梦想条件优先读取实体。 |
| Secret | 能发现、泄露、澄清秘密，并同步推进 P0 计数。 |
| Inventory | 能创建作品/信物/证据，支持转移和标记为线索。 |
| Authority | 凤姐/探春等人物能看到掌家权解释，分数不是固定写死。 |
| 分域声望 | 旧 `reputation` 兼容，新域能单独变化和读取。 |
| 后台 | 梦想页能查看并调试 P1 实体，不靠直接编辑 JSON。 |
| 连通性 | 连续模拟不报错；无 P1 实体时 P0 条件回退计数。 |

## 11. 实施顺序

1. 新增 `SocialIssueSystem`，先接后台创建/解决和梦想条件读取。
2. 新增 `SecretSystem`，接发现/泄露/澄清与 issue 联动。
3. 新增 `InventorySystem`，支持作品、信物、证据和转移。
4. 新增 `AuthoritySystem`，先用现有身份、家庭角色、关系、任务计数合成。
5. 新增 `ReputationDomainSystem`，兼容旧 `c.reputation`。
6. 后台 v2 梦想页新增 P1 实体区。
7. 跑 3 日模拟、梦想条件诊断和后台连通性检查。
