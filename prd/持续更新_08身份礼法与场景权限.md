## 2026-06-23 置顶补充：身份、关系、礼法统一方案

当前系统里有三组相近概念：

- `socialRank`：人物自己的社会位阶，如主子、仆从、外人。
- `relationInit.initType`：两人的名义关系，如父子、祖孙、主仆、朋友、夫妻。
- `relations` 四轴：两人的实际情分、信任、友谊、服从。

后续统一口径：不把三者硬合成一张表，而是新增一个上层解释器 `SocialContextSystem`，把“人物是谁、两人名义是什么、实际处得如何、当前行为合不合礼法”统一输出给任务、互动、场景权限、称谓、声望和 AI 使用。

### A. 四层模型

| 层级 | 核心问题 | 数据来源 | 例子 | 主要用途 |
|---|---|---|---|---|
| 身份位阶 | 这个人自身是什么身份 | `character.socialRank`、人生路径修正 | 至尊、长辈、平辈主子、晚辈主子、体面仆从、普通仆从、外人 | 任务权、场景权、基础礼法、声望分层 |
| 名分关系 | 这两人名义上是什么关系 | `relationInit.initType`、未来家族/契约系统 | 父子、母子、祖孙、夫妻、主仆、朋友、同事、亲戚 | 称谓、亲缘/姻亲、孝道/主仆语境、综合关系标签 |
| 实际关系 | 这两人实际处得怎么样 | `relations` 四轴 | 好感、信任、友谊、服从、综合分 | AI 候选、互动门槛、任务接受、告密/护短/记恨 |
| 礼法判定 | 当前行为是否合规矩 | 上述三层 + 行为 + 场景 + 见证者 | allowed / conditional / risky / forbidden | 互动提示、越界风险、阻止/放行、后果结算 |

关键原则：

- 身份位阶是个人属性，不等于两人关系。
- 名分关系是两人之间的社会称谓，不等于好感和信任。
- 实际关系是动态数值，不直接改写名分，但会影响是否愿意、是否冒险、是否服从。
- 礼法判定是运行时解释结果，不应自己再维护一套“家庭关系”。

### B. 统一接口目标

新增目标接口：

```js
SocialContextSystem.resolve(actorId, targetId, actionContext)
```

返回建议结构：

```js
{
  actorId,
  targetId,

  actorRank: 3,
  actorRankLabel: "晚辈主子",
  targetRank: 5,
  targetRankLabel: "普通仆从",
  rankRelation: "master_to_servant",
  rankRelationLabel: "主子→仆从",

  nominalRelation: "主仆",
  kinshipType: "无血缘",
  addressHint: "仆从称主子为姑娘/二爷/太太，语气恭顺。",

  relationScore: 52,
  affection: 52,
  trust: 41,
  friendship: 26,
  submissionActorToTarget: 0,
  submissionTargetToActor: 40,
  compositeRelationLabel: "心腹忠仆",

  protocolRole: "主子对仆从",
  protocolBehavior: "allowed",
  protocolReason: "主子吩咐仆从，合乎礼法。",
  riskLevel: "none"
}
```

`actionContext` 可选字段：

| 字段 | 含义 |
|---|---|
| `actionType` | `interaction / quest / scene_enter / dialogue / touch / economy / reputation` |
| `category` | 行为大类，如叙旧、传情、争执、差遣、请安。 |
| `templateId` | 互动或任务模板 ID。 |
| `sceneId` | 当前场景。 |
| `scenePrivacy` | `public / semi_private / private / forbidden`。 |
| `witnessIds` | 见证者。 |
| `contactType` | `none / touch / embrace / kiss / massage`。 |

### C. 与现有系统的迁移关系

| 当前接口 | 未来归属 | 迁移方式 |
|---|---|---|
| `IdentityProtocolSystem.getCharRank()` | 身份位阶层 | 保留，作为 `SocialContext` 输入。 |
| `IdentityProtocolSystem.getHierarchyRelation()` | 身份位阶关系推导 | 保留但改名倾向为 `getRankRelation()`，避免误解为家庭关系。 |
| `getRelationInfo()` | 名分关系 + 实际关系层 | 保留，`SocialContext` 读取 `initType` 与四轴。 |
| `IdentityProtocolSystem.resolveAddress()` | 称谓解释 | 改为优先读取 `SocialContext.nominalRelation`。 |
| `IdentityProtocolSystem.formatInteractionProtocol()` | 礼法判定 | 改为包装 `SocialContextSystem.evaluateProtocol()`。 |
| `InteractionScoreSystem.checkSoftLock()` | 实际关系门槛 | 继续负责综合分/轴要求，不和礼法规则混在一起。 |
| `InteractionSocialSystem.resolveRisk()` | 具体互动风险 | 继续处理身体接触、见证者、场景风险，读取 `SocialContext`。 |
| `QuestIssueSystem` | 任务权限 | 从直接读身份/关系，改为读取 `SocialContext`。 |
| `SceneAccessSystem` | 场景通行 | 从家庭角色、任务、邀请读取规则，补充读取 `SocialContext`。 |

### D. 声望模型

身份系统后续要接声望，但声望不应是一个全局总分。古代礼法语境下，“谁认可你”比“总声望多少”更重要。

建议每个人维护分域声望：

```js
reputation: {
  household: 0,
  masterCircle: 0,
  servantCircle: 0,
  clan: 0,
  outside: 0
}
```

| 声望域 | 含义 | 主子例子 | 仆从例子 |
|---|---|---|---|
| `household` | 家内总名声 | 管家有方、孝顺、体面 | 勤谨、少惹事、口碑稳 |
| `masterCircle` | 主子/长辈圈评价 | 得长辈宠信、能主持事务 | 得主子信任、可托付 |
| `servantCircle` | 仆从圈评价 | 宽厚或严苛带来的敬畏 | 有威信、嘴严、能护人或会告密 |
| `clan` | 宗族亲眷评价 | 贤孝、守礼、联姻体面 | 通常影响较小，特殊仆从可有 |
| `outside` | 外部声名 | 诗名、贤名、恶名 | 能干、风流、刁钻、心腹名声 |

主子和仆从的声望不走同一套解释：

- 主子高声望：贤孝、体面、管家有方、受长辈信任、能服众。
- 仆从高声望：忠心、能干、嘴严、得脸、有威信、差事稳。
- 某人可以在 `masterCircle` 高而在 `servantCircle` 低，例如得主子偏宠但被同伴嫉恨。
- 某人可以在 `servantCircle` 高而在 `masterCircle` 低，例如仆从中有人望但被主子疑忌。

### E. 后台统一方案

新后台建议拆为：

| 分组 | 页面 | 内容 |
|---|---|---|
| 关系 | 身份名分 | 人物身份位阶、两人名义关系、亲缘/姻亲/主仆分类。 |
| 关系 | 关系数值 | 好感、信任、友谊、服从、综合关系标签。 |
| 关系 | 礼法规则 | 行为类别 + 社会语境 -> allowed / conditional / risky / forbidden。 |
| 关系 | 声望规则 | 分域声望、身份差异化解释、声望变更来源。 |

当前 v2【身份系统】短期保留身份礼法相关表；场景配置和场景权限暂不作为常用后台入口，避免策划误用未完成能力。

身份页短期仍可保留两张表：

- 身份位阶：`socialRank -> rankLabel`。
- 两人身份关系枚举：由 `socialRank` 推导的 `rankRelation`，只用于任务权限和礼法初判。

但页面必须明确：`rankRelation` 不是【关系标签】里的家庭关系；家庭关系来自 `relationInit.initType`，未来由 `SocialContext` 汇总展示。

### F. 分阶段落地

| 阶段 | 目标 | 验收 |
|---|---|---|
| Phase 1 | 新增 `SocialContextSystem.resolve()`，只读现有数据，不改行为。 | 调试面板能看到同一对人物的身份位阶、名分关系、四轴、礼法结果。 |
| Phase 2 | 任务、互动、称谓改为读取 `SocialContext`。 | 任务权限、互动风险、称呼提示不再各算一套关系。 |
| Phase 3 | 场景权限读取 `SocialContext`，把场景权限配置并入场景配置页。 | 私宅、跟随、任务授权、邀请、擅闯解释一致。 |
| Phase 4 | 接入分域声望。 | 同一事件能分别影响主子圈、仆从圈、家内、外部声望。 |
| Phase 5 | 关系阈值、声望阈值触发记忆和剧情。 | 长模拟能解释“为什么此人得脸/失势/被嫉恨/被信任”。 |

## 1. 总目标

身份系统是大观园模拟的社会骨架。它不只是限制玩家，而是让越界、偏宠、请安、差遣、擅闯、告密这些戏剧行为有规则可循。

核心目标：

- 人物有清晰社会位阶与家庭身份。
- 互动、任务、移动和称谓都能读到同一套身份规则。
- 禁忌尽量转为风险和后果，只有极少数硬禁行为直接阻止。
- 场景权限让“谁能进谁的房、何时算擅闯、谁会驱离”稳定可解释。

## 3. 身份数据

主要配置在 `CONFIG.identityProtocolConfig`：

| 字段 | 含义 |
|---|---|
| `rankLabels` | 位阶展示名。 |
| `hierarchyLabels` | 身份关系展示名。 |
| `addressByInitType` | 称谓规则。 |
| `interactionCatMap` | 互动模板到礼法类别的映射。 |
| `rules` | 不同身份关系 + 互动类别的礼法行为。 |
| `riskyDefaults` | 默认逾矩风险配置。 |
| `contactTypeRules` | 亲密接触、公开接触等约束。 |

礼法行为建议保持四类：

| 行为 | 含义 |
|---|---|
| `allowed` | 正常允许。 |
| `conditional` | 条件允许，通常看关系、场景或状态。 |
| `risky` | 允许尝试，但失败或被看见有后果。 |
| `forbidden` | 硬禁止，慎用。 |

## 4. 场景权限数据

主要配置在 `CONFIG.sceneAccessConfig`：

| 字段 | 含义 |
|---|---|
| `sceneTypeLabels` | 场景类型展示。 |
| `privileges` | 不同家庭角色、仆从、访客的通行权。 |
| `invitations` | 临时邀请和任务放行。 |
| `guardRules` | 擅闯时谁有权驱离或记录。 |

放行来源优先级：

```text
公共场景
  -> 本家庭 / 主人 / 直属仆从权限
  -> 任务授权
  -> 跟随主子
  -> 邀请
  -> 特殊人生路径权限
  -> 拒绝进入或擅闯处理
```

## 5. 与其他系统的边界

- 关系系统只计算情分、信任、交情、服从，不决定礼法。
- 身份系统给互动和任务提供规则，不直接执行任务效果。
- 场景权限只管通行，不管人物为什么想去；动机仍由 AI、任务和需求产生。
- 逾矩后果通过状态、关系、记忆、叙事气泡和多人物反应结算。

## 6. 后续待办

- 正式区分“私密场景”“公开场景”“半公开场景”，让见证者风险更稳定。
- 场景守卫和告密链需要可配置：谁看见、告给谁、多久后发作。
- 任务授权应显示在人物调试面板中，解释为什么某人可以进某房。
- 身份成长线需要与人生路径合并，例如丫鬟晋升、主子降权、外客转客卿。
