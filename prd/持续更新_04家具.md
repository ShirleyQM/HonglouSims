## 1. 目标

把 `0616_家具模板优化.md` 中的“家具互动扩展表”整合进现有家具、行动队列、人物互动与在场反应系统。

核心变化：

- 家具不再只有一个默认“使用”动作，而是可以挂多个“家具互动动作”。
- 点击家具后，先打开家具动作菜单；选择动作后再入行动队列。
- 每个家具动作可以配置文案、时长、需求变化、物品产出、状态、技能经验、关系变化、戏剧触发。
- 在场人物会按性格/人物标签对家具动作产生反应，形成旁观、吐槽、帮忙、围观、打断等小戏剧。
- 家具动作与现有互动系统共享一部分规则：关系变化、状态变化、技能要求、礼法/身份、场景权限、埋点。

## 2. 现有系统盘点

### 2.1 家具模板

当前家具模板在 `js/config.js` 的 `furnitureTemplates` 中配置。

已支持字段：

| 字段 | 当前含义 |
|---|---|
| `name` | 家具名 |
| `category` | 家具类别，如 `desk`、`meal`、`bed` |
| `lifeLine` | 衣食住行闲分类 |
| `essential` | 是否基础生存家具，基础家具不卡技能 |
| `gridW/gridH` | 占地 |
| `entryOffset` | 入口格 |
| `icon/color` | 地图表现 |
| `needRestores` | 使用时按游戏分钟恢复需求 |
| `duration` | 默认使用时长 |
| `maxUsers` | 最大使用人数 |
| `skill/skillLevel/skillReq` | 技能门槛 |
| `stopWhenFull` | 需求满后提前停止 |
| `extraEffects` | 结束时附加效果，目前支持 `addState`、`skillXp` |

### 2.2 家具使用链路

当前链路：

1. `js/ui.js` 点击家具或家具入口格。
2. `makeFurnitureItem(inst)` 创建队列项。
3. `enqueueAction()` 入队。
4. `startPathToFurnitureStand()` 走到使用位。
5. `beginFurnitureUse()` 开始使用。
6. `updateAction()` 按 `needRestores` 持续恢复需求。
7. 结束后 `applyFurnEffects()` 结算 `extraEffects`。

这个链路稳定，可以继续复用。

### 2.3 当前不足

| 需求 | 当前状态 |
|---|---|
| 同一家具多个互动动作 | 已支持，模板可配置 `actions` |
| 家具动作菜单 | 已支持，多动作家具点击后选择动作 |
| 家具动作独立文案 | 已支持，队列和行动日志优先显示 action 文案 |
| 家具动作独立需求/状态/物品/技能奖励 | 部分支持，已支持需求、状态、技能经验、日志；物品/关系/记忆待接 |
| 在场人物标签反应 | 未支持 |
| 多人在场反应逐条触发 | 未支持 |
| 行动后戏剧触发 | 需要接入 `NarrativeBubbleSystem`、`MultiInteractSystem`、状态/记忆系统 |
| 后台配置复杂家具动作 | 当前后台只支持模板基础字段和需求恢复 |

## 3. 新数据结构建议

保留现有 `furnitureTemplates` 作为“家具实体模板”，新增 `actions` 字段描述可选互动。

示例：

```js
furnitureTemplates: {
  110: {
    name: '饭桌',
    category: 'meal',
    lifeLine: '食',
    essential: true,
    gridW: 2,
    gridH: 1,
    entryOffset: [1, 1],
    icon: '🍚',
    color: '#6a5040',
    duration: 2,
    maxUsers: 4,
    needRestores: [{ need: 'hunger', ratePerGameMin: 12 }],
    actions: [
      {
        id: 'eat_alone',
        name: '独自用膳',
        text: '一菜一汤，倒也自在',
        duration: 20,
        effects: [
          { type: 'need', need: 'hunger', delta: 30 },
          { type: 'need', need: 'mood', delta: 3 }
        ],
        tags: ['hunger', 'meal', 'solitude'],
        aiWeight: 1.2
      },
      {
        id: 'wait_to_eat',
        name: '等人共食',
        text: '菜要凉了，人怎么还不来',
        duration: -1,
        effects: [],
        tags: ['social', 'meal', 'waiting'],
        dramaHooks: ['late_guest_guilt']
      }
    ]
  }
}
```

兼容规则：

- 如果模板没有 `actions`，仍使用旧默认动作。
- 旧字段 `needRestores/duration/extraEffects` 自动视作一个 `default_use` 动作。
- 基础需求家具仍可由 AI 直接选默认动作，避免角色饿/困/脏时还要复杂决策。

## 4. 家具动作字段

| 字段 | 必填 | 说明 |
|---|---:|---|
| `id` | 是 | 动作 ID，在家具内唯一 |
| `name` | 是 | 展示名 |
| `text` | 否 | 开始/执行文案 |
| `duration` | 是 | 游戏分钟或秒，需要统一单位；建议使用游戏分钟 |
| `needRestores` | 否 | 持续恢复，沿用旧机制 |
| `effects` | 否 | 结束/即时结算效果 |
| `timing` | 否 | `start/end/tick`，默认 end |
| `tags` | 否 | AI、性格、戏剧识别用 |
| `skillReq` | 否 | 技能门槛 |
| `maxUsers` | 否 | 覆盖家具级最大使用人数 |
| `stopWhenFull` | 否 | 覆盖家具级满值中断 |
| `target` | 否 | 是否需要选择人物目标，如“留一块给人” |
| `reactionRules` | 否 | 在场人物标签反应 |
| `dramaHooks` | 否 | 触发叙事事件 |

## 5. 效果类型统一

建议把家具动作效果统一走 `CharacterEffectSystem.apply()`，减少家具系统单独判断。

当前已有：

- `state`
- `skillXp`
- `memory`
- `axis/relation`
- `need`

需要补齐或规范：

| 效果 | 用法 |
|---|---|
| `need` | 饥/倦/洁/闷/交游/心绪变化 |
| `state` | 添加状态，如偷嘴、感伤、迷糊 |
| `skillXp` | 诗词、女红、管理等经验 |
| `item` | 产出物品，如药汤、信笺、瓶花、绣品 |
| `relation` / `axis` | 与旁观者或目标人物变化 |
| `memory` | 触发记忆或新增记忆 |
| `bubble` | 头顶气泡 |
| `log` | 事件日志 |

## 6. 在场人物标签反应

`0616_家具模板优化.md` 的重点不是特定人物，而是“人物标签触发”。

建议复用现有人物性格配置：

- `charSpecialtyConfig.profiles[charId].aiTraits`
- `displayTraits`
- `traitMetadata`
- 旧 `CHAR_DEFAULT_TRAITS`

新增家具反应标签库时，优先映射到现有标签，避免另起一套。

示例映射：

| 文档标签 | 可映射现有标签 |
|---|---|
| 风雅 | `shuchi`、`qinggao` |
| 贪嘴 | 可新增 `tanchi` |
| 挑剔 | `kebo`、`qingjie` |
| 热心 | `haoke`、`zhongcheng` |
| 好事 | `haoke`、可新增 `haoshi` |
| 刻薄 | `kebo` |
| 沉闷 | `qinggao` 或新增 `chenmen` |
| 幽默 | `leitian` |
| 稳重 | `duanzheng` |
| 怯懦 | 可新增 `qienuo` |
| 精明 | 可新增 `jingming` |
| 痴情 | `duoqing` |
| 莽撞 | 可新增 `mangzhuang` |

反应规则结构：

```js
reactionRules: [
  {
    matchTraits: ['shuchi', 'qinggao'],
    priority: 10,
    text: '{observer}凝神听着，微微点头。',
    effects: [
      { type: 'need', target: 'observer', need: 'mood', delta: 3 },
      { type: 'axis', target: 'actor', axis: 'friendship', delta: 2 }
    ]
  },
  {
    matchTraits: ['kebo'],
    priority: 5,
    text: '{observer}皱眉道：这也算弹琴？',
    effects: [{ type: 'need', target: 'actor', need: 'mood', delta: -3 }]
  }
]
```

触发范围：

- 默认只检查同一 `sceneId`。
- 可配置 `radius`，比如琴声、药味、酒令可影响邻近范围。
- 每个旁观者只命中最高优先级一条。
- 多个旁观者可分别触发，但要限制每次最多 2-3 条日志/气泡，避免刷屏。

## 7. 与现有互动系统的边界

家具动作分三类：

| 类型 | 说明 | 处理方式 |
|---|---|---|
| 单人家具动作 | 独自用膳、抚琴自娱、洗手净面 | 走家具系统 |
| 家具引发旁观反应 | 抚琴时路人驻足、药味飘散 | 家具系统 + reactionRules |
| 需要明确目标人物 | 邀人共饮、并坐聊天、推人荡秋千 | 转成互动系统或家具动作选择目标后执行 |

建议：

- “邀人共饮”“并坐聊天”这类强社交动作，最终应生成一个 `interaction` 队列项，或生成 `furnitureInteraction` 队列项但内部持有 `targetCharId`。
- 初期先做单人家具动作和旁观反应。
- 第二阶段再做带目标人物的家具动作。

## 8. 后台配置迭代

当前后台 `家具模板` 只支持基础字段。

需要新增：

1. `actions` JSON 编辑区。
2. 常用动作表单：
   - 动作 ID
   - 名称
   - 文案
   - 时长
   - 标签
   - 技能要求
   - 需求变化
   - 状态/技能/物品/关系效果
3. `reactionRules` JSON 编辑区。
4. 一键从旧模板生成默认动作。
5. 校验：
   - action id 不重复
   - need key 存在
   - state id 存在
   - skill id 存在
   - trait id 存在

## 9. 实现路径

### 阶段 1：动作模型兼容层

状态：已完成。

目标：不改变现有体验，但让系统能理解家具多动作。

改动：

- 新增 `getFurnitureActions(tpl)`。
- 没有 `tpl.actions` 时生成 `default_use`。
- `makeFurnitureItem(inst, action)` 支持传入具体 action；旧调用不传 action 时自动取默认/首个 action。
- `beginFurnitureUse()` 接收 `item.furnitureAction`，优先使用 action 的 `duration/needRestores/effects/text/tags`。

验收：

- 旧家具全部还能正常使用。
- 默认动作行为与现在一致。

### 阶段 2：家具动作菜单

状态：已完成基础版，2026-06-17 已改为复用人物互动环形菜单。

目标：玩家点击家具后能选择具体动作。

改动：

- `handleMapClick()` 点击家具时：
  - 如果只有一个默认动作，沿用直接入队。
  - 如果多个动作，打开和人物互动同款的环形菜单。
- 菜单展示：
  - 动作名
  - hover 详细描述
  - hover 内包含时长、主要需求变化、技能门槛/不可用原因
- 当前版本点击不可用动作会走 `canUseFurniture()` 校验并冒气泡提示；灰态展示待补。

验收：

- 饭桌可选“独自用膳/挑食抱怨”；“等人共食”归入目标人物动作阶段。
- 技能不足或身份不符时不入队并提示。

### 阶段 3：动作效果统一

状态：基础版已完成。

目标：把家具动作效果接入统一效果系统。

改动：

- `applyFurnEffects()` 已处理 action-level `effects`。
- 已支持 `need/state/skillXp/log`。
- 待支持 `relation/memory/item/bubble`。

验收：

- “偷尝一块”能加饥、减闷，并概率触发偷嘴状态。
- “抄写诗文”能加诗词经验或心绪。

### 阶段 4：AI 接入

状态：部分完成。

目标：NPC 会选择不同家具动作，而不是只选择家具。

当前行为：AI 候选池已从“家具实例”扩展为“家具实例 + action”。候选 key 形如 `furn:4003:tea_alone`，入队时会携带 `furnitureAction`。

改动：

- `buildCandidatePool()` 中家具候选从“家具实例”扩展为“家具实例 + action”。（已完成）
- 候选 key 从 `furn:instanceId` 变为 `furn:instanceId:actionId`。（已完成）
- action tags 参与 `calcDemandFactor()`、性格权重、戏剧权重。（已完成基础接入）
- 基础需求紧急时仍偏向最直接的 action。（已完成基础接入）

验收：

- 饿的时候选“独自用膳/偷尝一块”。
- 闷的时候选“抚琴自娱/赏花发呆/翻闲书”。
- 心绪低时选“躲被痛哭/抄经静心/赏月独坐”。

### 阶段 5：在场人物反应

状态：基础版已完成。

目标：家具动作产生场景戏剧。

改动：

- 新增 `FurnitureReactionSystem`。（已完成）
- 在 `furniture:use_started` 时触发。（已完成）
- 查找同场景旁观者。（已完成）
- 按 `furnitureReactionConfig.rules` 优先级匹配 action tags、动作 ID、人物性格、关系范围。（已完成基础接入）
- 应用轻量效果并发日志/气泡/埋点。（已完成基础接入）

验收：

- 黛玉/宝玉听琴会驻足评价。
- 刻薄者会吐槽。
- 热心者会帮忙。
- 多人在场时最多输出若干条，不刷屏。

### 阶段 6：目标人物型家具动作

状态：未开始。

目标：实现“邀人共饮”“并坐聊天”“推人荡秋千”等。

改动：

- action 增加 `targetMode: 'selectChar' | 'nearbyChar' | 'auto'`。
- 点击动作后选择目标人物，或 AI 自动选目标。
- 可复用互动系统：
  - 轻量版本：家具动作内部带 `targetCharId`。
  - 完整版本：转为 `interactionTemplate` 或临时 interaction。

验收：

- 茶案可邀人共饮。
- 竹榻可并坐聊天。
- 秋千可推人荡秋千。

## 10. 推荐首批落地内容

不要一次把文档里所有家具动作都做完。建议先选 5 个家具覆盖主要机制：

1. 饭桌：独自用膳、挑食抱怨。
2. 红木书案：抄写诗文、翻闲书。
3. 琴台：抚琴自娱、弹错走调。
4. 花圃：赏花发呆、埋花葬花。
5. 茶案：独坐品茗、邀人共饮（第二阶段目标动作）。

这 5 个能覆盖：

- 基础需求恢复
- 心绪/意趣变化
- 技能门槛
- 技能经验
- 状态触发
- 旁观反应
- 目标人物动作

## 11. 风险与设计决定

### 家具类别标签不能替代动作语义

2026-06-17 发现：茶案 `category=table` 曾被 AI 继承为 `hunger/xujiu/食`，但 `独坐品茗` 实际只恢复 `闷/心绪`。结果是黛玉等角色会反复去茶案，甚至多人围在同一家具附近。

已修正：

- AI 判断需求相关性以 action 的 `needRestores/tags` 为主。
- `table` 不再作为饥饿紧急家具。
- 同一家具已有使用者/排队者/占位者时，家具候选会按拥挤度降权。
- 茶案动作可通过 `aiWeight` 单独调低，不影响玩家手动选择。

### 单位统一

当前家具模板 `duration` 实际按秒使用；文档写的是分钟。

建议：

- 内部统一仍用秒，避免改动行动队列。
- 配置后台显示“游戏分钟/现实秒”需要注明。
- 或新增 `durationGameMin`，由系统转换成现实秒。

### AI 复杂度

多动作后候选池会膨胀。

建议：

- 每个家具最多参与 2-3 个高权重动作。
- 低权重动作只在玩家菜单里出现。
- AI 候选池保留 `cacheMaxSize` 截断。

### 反应刷屏

多人在场反应容易刷日志。

建议：

- 每次家具动作最多 2 条气泡、3 条日志。
- 同一旁观者对同类 action 有冷却。

### 与互动系统重复

“邀人共饮”和 interactionTemplates 的“品茗/对酌”会重复。

建议：

- 家具动作负责场景入口和家具占位。
- interactionTemplates 负责人与人关系、礼法和对白。
- 需要两人共同执行的家具动作，最终应桥接到互动系统。

## 12. 当前结论

现有系统可以平滑扩展，不需要推倒重写。

已落地路线：

1. 家具模板 `actions` 兼容层。
2. 玩家家具动作菜单。
3. action-level `duration/needRestores/effects/text` 接入行动链路。
4. 首批书案、饭桌、茶案、琴台、花圃动作配置。

下一步优先级：

1. 目标人物型家具动作。
2. 为家具动作补 `aiWeight` 调参建议和后台结构化表单编辑。
3. 家具反应规则继续接 LLM 文案、打断/帮忙/围观等更强行为。

这样每一步都能独立验收，也不会影响现有“点家具就能解决基础需求”的核心循环。
