## 1. 总目标

基础需求系统是人物日常行为的底层压力源。它回答：人物现在缺什么、缺到什么程度会改变行为、满足或失衡后会留下什么状态。

六需求当前口径：

- 饥饿 `hunger`
- 精力 `energy`
- 意趣 `fun`
- 洁净 `hygiene`
- 交游 `social`
- 心绪 `mood`

## 3. 数据口径

### 3.1 需求定义

| 字段 | 含义 |
| --- | --- |
| `key` | 需求 ID。 |
| `name` | 前后台展示名。 |
| `defaultGrow` / `defaultDecay` | 默认恢复和衰退系数。 |
| `stateBands` | 需求分档状态。 |
| `baseNeedCoeffs` | 人物个体差异，含 `min/max/grow/decay`。 |

运行时只认 `baseNeedCoeffs` 作为人物基础系数；旧字段如 `needCoeffs` 只作为兼容读取，不应继续写入。

### 3.2 状态定义

| 字段 | 含义 |
| --- | --- |
| `name` / `desc` | 展示名和说明。 |
| `duration` | 持续游戏分钟。 |
| `category` | mood、need、social、task 等分类。 |
| `polarity` | positive、negative、mixed、neutral。 |
| `conflictGroup` | 同组状态互斥。 |
| `needMods` | 需求系数修正。 |
| `aiModifiers` | AI 标签权重修正。 |
| `blockedSkills` | 阻塞技能或行动能力。 |
| `traitEffects` | 特定性格追加效果。 |

## 4. 运行流程

```text
游戏时间推进
  -> 自然衰退需求
  -> 需求变化事件
  -> CoreNeedSystem 计算交游/心绪牵引
  -> NeedStateSystem 同步分档和组合状态
  -> StateSystem 更新状态时长、冲突和容量
  -> Utility AI 读取需求、状态、性格、家具恢复
```

## 5. 与起居和 Utility AI 的关系

- 需求不是只有低了才救火，也会参与一天节律。
- 起居里的需求活动通过 `completeBy.categories/tags` 匹配家具/社交候选。
- 严重需求危机优先于普通起居和闲逛。
- 新起居有块时覆盖旧作息兜底；空白时段继续读取旧基础作息/日程兜底。

## 7. 饮酒状态与健康边界

饮酒不是第七个基础需求，不进入饥饿/精力/洁净/交游/意趣/心绪六需求循环。它作为运行时负荷影响状态和健康：

- `AlcoholSystem` 维护人物当日饮酒剂量和当前醉意负荷。
- 状态层使用 `drunk_state` 互斥组，包含微醺、醺然、酒醉、酩酊、宿醉等。
- 普通小酌不应直接给 `drunk`，而是先增加醉意负荷，由阈值决定是否进入微醺。
- 移动禁用只应出现在重度醉态，如 `deadDrunk`；普通 `drunk` 更适合表现为移动变慢、办事失误率上升。
- 健康扣减来自当日累计过量和长期频繁纵饮，不由单次小酌直接触发。

详细设计见：[持续更新_31_酒桌与饮酒系统.md](持续更新_31_酒桌与饮酒系统.md)。
