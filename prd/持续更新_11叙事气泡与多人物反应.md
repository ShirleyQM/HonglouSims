# 叙事气泡与多人物反应 PRD

## 2026-06-22 置顶补充：性格 × 需求 × 状态驱动

当前叙事气泡仍偏粗糙：`demandBubbles` 只在低需求事件里按单个性格筛选，`interactionStateBubbles` 只按状态出短句，记忆、冲突、旁观反应也各走各的触发分支。下一版要把气泡升级为“人物上下文驱动的表现层”：先合成人物当下的性格、需求、状态、行动、关系和场景，再统一打分选择气泡。

原则：

- 叙事气泡仍是表现层，不成为 AI 主脑，不阻塞玩家操作。
- 性格、需求、状态必须一起参与，而不是各写一套孤立文案。
- 需求提供“为什么此刻想说”，状态提供“情绪温度”，性格提供“用什么口吻说”。
- 同一个状态在不同性格上应有不同表达；同一个低需求在不同状态下也应有不同表达。
- 确定性模板必须可独立运行；LLM 只能在规则命中后润色文本，不能决定效果和条件。

### A. 当前问题诊断

| 问题 | 当前表现 | 改造目标 |
|---|---|---|
| 触发碎片化 | 低需求、互动状态、记忆、冲突分别处理 | 统一进入 `BubbleContext -> Candidate -> Arbiter` 流程 |
| 性格条件太薄 | 只支持 `condition.trait` 单一匹配 | 支持 `traitsAny` / `traitsAll` / `traitsNot` 与专属性格 |
| 需求只看低值 | 主要监听 `need:critical` | 支持低、高、变化趋势、主导需求和需求分档状态 |
| 状态只做文案触发 | `stateId` 命中就随机出一句 | 状态参与强度、口吻、优先级和互斥判断 |
| 缺少解释 | 只知道显示了什么，不知道为什么 | 每次气泡记录驱动来源：性格 / 需求 / 状态 / 事件 |
| 记忆浮现割裂 | `memorySurfaces` 只在进场时独立触发 | 记忆作为候选来源，和性格、状态、场景一起判断 |
| 容易刷屏 | 依靠模块级 burst 和文本去重 | 增加规则级、人物级、状态级冷却与每日上限 |

### B. 运行结构

新增统一流程：

```text
事件或定时扫描
  -> buildBubbleContext(char, event)
  -> collectBubbleCandidates(context)
  -> scoreBubbleCandidate(candidate, context)
  -> arbitrateBubble(candidates, context)
  -> resolveText(rule, context)
  -> showBubble()
  -> emit bubble:candidate / bubble:rejected / bubble:show
```

`BubbleContext` 至少包含：

| 字段 | 来源 | 用途 |
|---|---|---|
| `charId` / `name` | 人物 | 模板填充与日志 |
| `traits` | `getCharTraits()` / `TraitEffectSystem` | 口吻、偏好、规则条件 |
| `specialties` | `charSpecialtyConfig.profiles[].specialties` | 角色个性化梗与稀有气泡 |
| `needs` | `CoreNeedSystem` / `needDefs` | 当前值、分档、最低需求、变化趋势 |
| `states` | `activeStates` / `stateDefs` | 情绪、身体、社交、任务状态 |
| `scene` | `sceneAt()` / `sceneId` | 场景限定与氛围 |
| `action` | 当前 AI / 行动队列 | 正在做什么、是否被打断 |
| `target` | 互动对象 / 观察对象 | 对谁说、因谁触发 |
| `relation` | 关系系统 | 关系锚点与称呼风格 |
| `memories` | 人物记忆 | 最近记忆、同场景记忆、同目标记忆、记忆情绪标签 |
| `time` | 游戏时间 | 时辰、白天/夜间、日程段 |
| `event` | 触发事件 | 本次候选来源 |

### C. 新配置协议

在 `narrativeBubble` 下新增 `drivenRules`，逐步替代分散硬编码；旧的 `demandBubbles`、`interactionStateBubbles`、`memorySurfaces` 继续兼容，并作为候选来源接入统一仲裁。

```js
narrativeBubble: {
  drivenRules: [
    {
      id: 'hygiene_aijie_low',
      name: '爱洁遇低洁净',
      module: 'demand',
      trigger: { type: 'need_band', needKey: 'hygiene', band: 'low' },
      conditions: {
        traitsAny: ['aijie', 'qingjie'],
        specialtiesAny: [],
        needs: [{ key: 'hygiene', op: '<=', value: 45 }],
        statesAny: ['needHygieneLow', 'needHygieneCritical'],
        statesNot: ['sleeping'],
        sceneTypes: [],
        actionTagsAny: [],
        relation: null,
        timeWindows: []
      },
      score: {
        base: 60,
        needWeight: 1.2,
        stateWeight: 1.1,
        traitWeight: 1.15,
        priority: 20,
        cooldownGameMin: 30,
        dailyMaxPerChar: 4
      },
      style: 'exclaim',
      icon: 'clean',
      texts: [
        '这衣上怎么沾了灰？',
        '这样腌臜，实在坐不住。'
      ],
      effects: []
    }
  ]
}
```

字段约定：

| 字段 | 含义 |
|---|---|
| `trigger.type` | `event`、`scan`、`need_band`、`state_add`、`state_refresh`、`interaction`、`furniture`、`idle` |
| `conditions.traitsAny/All/Not` | 通用性格条件 |
| `conditions.specialtiesAny/All/Not` | 人物专属性格条件 |
| `conditions.needs` | 需求阈值，支持 `<`、`<=`、`>`、`>=`、`between` |
| `conditions.statesAny/All/Not` | 当前状态条件 |
| `conditions.stateCategory` | 按 `mood`、`needBand`、`social`、`task` 等状态分类匹配 |
| `conditions.statePolarity` | `positive`、`negative`、`mixed`、`neutral` |
| `conditions.actionTagsAny` | 当前行动或家具标签 |
| `conditions.relation` | 与目标的关系轴、区间或关系阶段 |
| `conditions.memoryTagsAny` | 记忆标签，例如温情、创伤、传情、争执 |
| `conditions.memoryScene` | 是否要求记忆发生在当前场景 |
| `conditions.memoryTarget` | 是否要求记忆关联当前互动对象 |
| `score.base` | 规则基础分 |
| `score.needWeight` | 需求压力越强，分数越高 |
| `score.stateWeight` | 状态强度和负面程度加权 |
| `score.traitWeight` | 性格匹配加权，不得盖过需求和关系锚点 |
| `score.priority` | 同分时优先级，数值越小越优先 |
| `effects` | 只允许轻量、白名单效果；默认不配置 |

### D. 打分和仲裁

候选分数建议：

```text
score =
  base
  × triggerStrength
  × needPressure
  × stateIntensity
  × traitAffinity
  × relationRelevance
  × noveltyFactor
  × cooldownFactor
```

仲裁规则：

- 单人物同一时间最多显示 1 个叙事气泡。
- 全局同时显示仍受 `settings.maxOnScreen` 控制。
- 优先级：危机状态 > 极低基础需求 > 互动结果 > 任务/家具反馈 > 性格风味 > 闲置氛围。
- 同一人物同一 `rule.id` 必须有冷却。
- 同一人物同一 `stateId` 的状态气泡应有冷却，避免状态刷新刷屏。
- 玩家主动操作不被气泡频控阻塞；频控只影响是否展示文案。

### E. 文案模板变量

统一支持以下占位：

| 占位 | 含义 |
|---|---|
| `{name}` | 当前人物名 |
| `{target}` / `{other_name}` | 目标人物名 |
| `{trait}` / `{trait_label}` | 命中的主要性格 |
| `{specialty}` / `{specialty_label}` | 命中的专属性格 |
| `{need_name}` / `{need_value}` / `{need_band}` | 需求名称、数值、分档 |
| `{state_name}` / `{state_desc}` | 状态名称和描述 |
| `{scene}` | 当前场景 |
| `{action}` | 当前行动 |
| `{time_period}` | 当前时辰段 |
| `{relation_label}` | 与目标的关系阶段 |

### F. 首批规则样例

| 场景 | 驱动 | 气泡方向 |
|---|---|---|
| 爱洁 + 洁净低 + 低洁净状态 | `aijie/qingjie` + `hygiene <= 45` + `needHygieneLow` | 对脏污敏感，主动表达不适 |
| 邋遢 + 洁净低 + 心绪尚可 | `lata` + `hygiene <= 35` + 非负面心绪 | 觉得还能忍，不急着处理 |
| 多愁 + 心绪低 + 夜间/风景场景 | `duochou` + `mood <= 45` + `melancholy/ganshang` | 忧思、感景、想独处 |
| 才华 + 意趣低 + 书房/园景 | `caihua` + `fun <= 45` + 无强负面状态 | 想读书、写诗、赏景 |
| 热络 + 交游低 + 同场景有人 | `reluo/haoke` + `social <= 45` + 无冲突状态 | 想搭话或邀人同行 |
| 孤僻 + 交游低 + 人多 | `gupi/qinggao` + `social <= 45` + 人群密度高 | 不是想社交，而是想避开喧闹 |
| 嗜睡 + 精力低 + 夜间 | `shishui` + `energy <= 50` + 夜间 | 比普通人更早、更自然地困倦 |
| 多情/女性社交大类 + 对方负面状态 | `duoqing/female_charm` + 目标 `negative` 状态 | 想安慰，但权重不能盖过关系锚点 |
| 进场 + 同场景旧记忆 + 当前状态负面 | `scene:entered` + `memoryTag` + `melancholy/ganshang` | 触景生情，旧事被当前情绪放大 |
| 进场 + 温情记忆 + 乐天/厚道 | `scene:entered` + `温情` + `leitian/houdao` | 想起旧事但表达更温暖、释然 |
| 遇见目标 + 传情记忆 + 心绪波动 | `interaction` + `memory.with = target` + `heartFlutter/secretCrush` | 不是随机害羞，而是由关系记忆触发 |

### G. 事件接入

首版需要监听：

- `need:critical`：极低需求。
- `need:band_changed`：需求分档变化。
- `state:add` / `state:refresh`：状态新增或刷新。
- `interaction:state` / `interaction:complete`：互动结果。
- `scene:entered`：进入场景；用于同场景记忆浮现。
- `furniture:use_started` / `furniture:use_complete`：家具行动。
- `ai:decision`：AI 选择了某行动，用于低频解释气泡。
- `time:tick`：低频扫描闲置、场景和时辰氛围。

记忆浮现接入规则：

- 旧 `memorySurfaces` 先不删除，迁移期由系统转成 `module: 'memory'` 的候选。
- 记忆不再只按“进入场景 + 概率”直接显示，而要参与统一仲裁。
- 同一条记忆可因性格和当前状态出现不同文案：多愁偏感伤，乐天偏释然，敏感偏反复咀嚼，热络偏想找人说。
- 记忆候选必须带上 `memoryId/memoryTag/memoryWith/memoryScene`，便于调试和去重。
- 同一人物同一记忆有冷却；同一场景短时间内最多浮现 1 条记忆，避免进出门刷屏。

### H. 后台与调试

后台【叙事气泡】需要支持：

- `drivenRules` 表格维护：触发、条件、打分、冷却、文案、启停。
- 条件编辑器优先提供结构化控件，保留高级 JSON。
- 每条规则显示“影响系统：性格 / 需求 / 状态 / 关系 / 场景”。
- 记忆规则显示“记忆标签 / 关联人物 / 关联场景 / 最近触发时间”。
- 调试面板展示最近 50 条候选：命中、被拒、最终展示及原因。
- 人物详情页展示“最近气泡驱动”：例如 `多愁 + 心绪低 + 感伤`。

埋点：

| 事件 | 字段 |
|---|---|
| `bubble:candidate` | `ruleId`、`charId`、`score`、`traitIds`、`needKeys`、`stateIds`、`sourceEvent` |
| `bubble:rejected` | `ruleId`、`charId`、`reason`、`score` |
| `bubble:show` | `ruleId`、`charId`、`module`、`text`、`drivers`、`memoryId` |

### I. 验收标准

- 运行 3 游戏日，气泡日志中至少 80% 可追溯到明确的 `traitIds`、`needKeys` 或 `stateIds`。
- 爱洁人物在洁净偏低且有低洁净状态时，优先出现洁净相关文案，而不是泛泛低需求文案。
- 同样是 `melancholy`，多愁人物、乐天人物、热络人物的文案池有明显差异。
- 同样是交游低，热络人物倾向“想找人说话”，孤僻/清高人物倾向“嫌闹或想清净”，不能统一写成“寂寞”。
- 同样进入一处有旧记忆的场景，多愁/敏感人物更容易触景生情，乐天/厚道人物更容易出现温和回忆或释然文案。
- 状态刷新、低需求持续、多人同场景时不刷屏；同一人物同一规则在冷却内不重复展示。
- LLM 关闭时游戏仍完整可玩；LLM 开启时只润色已命中规则，不引入未校验效果。

## 1. 总目标

叙事气泡和多人物反应负责让模拟结果“被看见”。它们不做底层决策，而是把需求、状态、冲突、旁观、情绪传染、家具使用和性格行为转成短句、反应和轻量效果。

目标：

- 人物有内心独白和现场反应。
- 旁观者能对互动、状态、擅闯、家具行为做出反应。
- 情绪和冲突可以在人群中传播。
- 气泡可模板化，也可发出 LLM 请求，但不绕过底层规则。

## 3. 配置口径

### 3.1 `narrativeBubble`

| 子表 | 用途 |
|---|---|
| `demandBubbles` | 需求低时的内心或说话气泡。 |
| `interactionStateBubbles` | 互动结果状态短句。 |
| `conflictTheaters` | 双人冲突小剧场。 |
| `memorySurfaces` | 记忆浮现规则。 |
| `settings.llm` | 气泡 LLM 事件或 API 配置。 |

### 3.2 `multiInteractConfig`

后台【互动模板和多人互动】已把多人规则拆成表格，不再只编辑完整 JSON。

#### 3.2.1 总开关和基础参数

| 字段 | 含义 |
|---|---|
| `masterEnabled` | 多人规则总开关。 |
| `maxObserverBubblesPerEvent` | 同一事件最多触发几个旁观气泡，防止刷屏。 |
| `contagionCooldownGameMin` | 情绪传染冷却，单位游戏分钟。 |

#### 3.2.2 状态标签映射 `statusTags`

`statusTags` 把策划叙事标签映射到真实状态 ID。

```js
statusTags: {
  感伤: 'ganshang',
  暴怒: 'angry',
  开心: 'elated'
}
```

用途：

- 规则可写“感伤”“开心”这类策划语言。
- 实际执行仍落到 `stateDefs` 的状态 ID。
- 如果状态 ID 不存在，规则应视为无效或只发提示不结算状态。

#### 3.2.3 多人 Prompt 模板 `promptTemplates`

`promptTemplates` 给 LLM 或模板对白提供多人语境。

| 字段 | 含义 |
|---|---|
| key | prompt 标签，例如 `mediate_conflict`。 |
| text | 模板文本，可使用 `{observer.name}`、`{observed.name}`、`{target.name}`。 |

LLM 只负责生成文本，不直接决定效果。效果仍由 `observerReactions[].reactionType` 和 `reactionParams` 执行。

#### 3.2.4 旁观/连锁反应 `observerReactions`

每条 `observerReactions` 是一个“有人看见某事后做什么”的规则。

| 字段 | 含义 |
|---|---|
| `id` | 规则 ID。 |
| `name` | 规则名。 |
| `priority` | 优先级，数值越小越优先。 |
| `cooldownGameMin` | 冷却，单位游戏分钟。 |
| `observerCharId` | 谁会旁观/反应。 |
| `observedCharId` | 被观察的人，可空表示任意人。 |
| `triggerSignal` | 触发信号，例如 `interaction`、`status_gained`、`action_tag`。 |
| `triggerInteractionCategory` | 触发的大互动类别，例如 `zhengchi`、`chuanqing`。 |
| `triggerStatus` | 触发状态，例如 `ganshang`。 |
| `triggerActionTag` | 触发行动标签，例如 `娱乐`。 |
| `triggerTargetRole` | 目标身份条件，例如 `仆从`。 |
| `reactionType` | 反应类型，决定执行器。 |
| `reactionParams` | 反应参数 JSON。 |
| `bubbleTexts` | 旁观气泡文本数组。 |
| `llmConfig` | LLM 开关、promptTag、fallbackTexts。 |
| `additionalEffects` | 追加状态/关系等效果。 |

常见 `reactionType`：

| 类型 | 含义 | 典型参数 |
|---|---|---|
| `send_bubble` | 只发旁观气泡。 | `bubbleStyle` |
| `force_interaction` | 观察者插入一次互动。 | `interactionTemplateId`、`interruptionBehavior` |
| `issue_quest` | 观察者下发任务。 | `questTemplateId` |
| `add_status` | 给观察者或被观察者加状态。 | `target`、`statusId` |
| `boost_action` | 临时提高某类行动权重。 | `actionTag`、`weightMultiplier`、`durationGameMin` |

触发流程：

```text
事件发生
  -> 扫描 observerReactions
  -> 匹配 observer / observed / triggerSignal / category / status / tag
  -> 按 priority 和 cooldown 仲裁
  -> 执行 reactionType
  -> 产生气泡、互动、任务、状态或 AI 权重
```

#### 3.2.5 情绪传染 `emotionContagion`

`emotionContagion` 是“某人状态影响附近人”的规则。

后台当前按每条规则的一层字段展开；复杂字段可保留 JSON。

建议字段口径：

| 字段 | 含义 |
|---|---|
| `id` / `name` | 规则标识和展示名。 |
| `sourceStatus` | 来源状态。 |
| `targetStatus` | 传播到旁人的状态。 |
| `radius` | 影响半径。 |
| `probability` | 基础概率。 |
| `relationFactor` | 关系越近是否更容易传染。 |
| `traitsBoost` / `traitsResist` | 哪些性格更易受影响或更抗拒。 |
| `cooldownGameMin` | 冷却。 |

情绪传染不得绕过状态容量、冲突组和状态免疫规则。

## 4. 事件来源

常见信号：

- `interaction`：社交互动发生。
- `interaction:state`：互动结果状态。
- `status_gained`：人物获得状态。
- `action_tag`：行动或家具标签。
- `scene_enter`：进入场景。
- `scene_enter_unauthorized`：擅闯或无权限进入。
- `time:tick` / `time:hour` / `time:day`：定时扫描。

## 5. 边界规则

- 气泡系统可以展示、轻量提示和应用配置化效果，但不应成为 AI 主脑。
- 多人物反应产生任务时，应走 `QuestIssueSystem` 或明确的任务模板。
- LLM 返回只能作为文字，不能直接携带任意 JS 或未校验效果。
- 旁观反应必须有冷却和上限，避免同一事件被同场景所有人重复刷屏。

## 6. 后续待办

- 把 `observerReactions` 按“告密 / 劝解 / 起哄 / 护短 / 接管任务”拆成后台分组。
- 情绪传染需要更明确的状态白名单和强度解释。
- 冲突剧场应接入关系四轴阶段名，而不只看综合分。
- LLM 气泡要增加请求日志和失败回退统计。
