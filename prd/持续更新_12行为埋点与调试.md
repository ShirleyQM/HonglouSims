# 行为埋点与调试 PRD

> 更新时间：2026-06-21
> 来源：当前代码、模拟报告和测试脚本。
> 主要代码：`js/systems/behavior-telemetry.js`、`js/ai.js`、`js/systems/quest.js`、`reports/`、`scripts/`。

## 1. 总目标

行为埋点系统用于回答“为什么 NPC 这么做”和“这套配置跑多天后是否合理”。

它服务三类问题：

- AI 决策解释：候选从哪里来、为什么被拒、为什么选中。
- 任务链排查：为什么没接、为什么阻塞、为什么完成质量低。
- 长期平衡：某人物几天内是否一直吃饭、一直社交、一直卡在某房。

## 3. 事件数据口径

通用事件应包含：

| 字段 | 含义 |
|---|---|
| `type` | 事件类型。 |
| `charId` | 主体人物。 |
| `ts` / `gameDay` / `gameHour` | 现实与游戏时间。 |
| `data` | 事件特定数据。 |

AI 决策事件建议包含：

| 字段 | 含义 |
|---|---|
| `channel` | fast、slow、urgent 等。 |
| `providerStats` | 各候选来源 checked、accepted、rejected。 |
| `picked` | 选中候选。 |
| `topCandidates` | Top 候选及权重。 |
| `scoreFactors` | 评分因子。 |
| `scoreHints` | 最关键解释文本。 |

## 4. 调试入口

- 浏览器本地存储：`dgy_behavior_telemetry_v1`。
- `BehaviorTelemetry.query(options)`：按人物、类型和时间查询。
- `BehaviorTelemetry.summarize(charId, options)`：人物行为摘要。
- `BehaviorTelemetry.download(charId)`：导出 JSON。
- `reports/`：保存模拟报告、布局 demo、检查点。

## 4.1 管理任务与秩序埋点

为了后续观察“管事是否顺”“任务造成了什么影响”，任务生命周期事件会额外携带秩序上下文。

覆盖事件：

- `quest:issued`
- `quest:acceptance_checked`
- `quest:accepted`
- `quest:started`
- `quest:blocked`
- `quest:progress`
- `quest:completed`
- `quest:failed`
- `quest:declined`
- `quest:expired`
- `order:impact`
- `character:effect`
- `economy:quest_cost`
- `economy:quest_reward`
- `economy:quest_fine`
- `punishment:applied`
- `punishment:skipped`
- `punishment:money_pending`

新增字段口径：

| 字段 | 含义 |
|---|---|
| `orderAffairId` | 秩序事务类型，如 `ritual`、`study`、`routine`、`account`、`cleaning`、`message`。 |
| `orderAffairLabel` | 事务中文名。 |
| `orderOwnerId` / `orderOwnerName` | 当前按职位/权柄推导出的建议归口人。 |
| `orderIssuerOwnsAffair` | 下发者是否在该事务归口内。 |
| `orderIsManagementTask` | 是否按管理类任务强制保留 telemetry。 |
| `orderTargetFitScore` | 执行者对该事务的适配分，读取技能、身份位阶、职位归口。 |
| `orderTargetFitReasons` | 适配分解释。 |
| `orderMetrics` | 执行前的秩序参数快照。 |
| `orderDelta` | 本次任务对秩序参数的估算影响，不直接改游戏数值。 |
| `issuerTraits` / `assigneeTraits` | 下发者/执行者的性格与专属性格快照，用于解释任务接受和执行差异。 |

任务经济事件字段：

| 字段 | 含义 |
|---|---|
| `templateId` / `quest` | 任务模板与任务名。 |
| `issuerId` / `assigneeId` | 下发者与执行者。 |
| `familyId` / `fromFamilyId` / `targetFamilyId` | 发生扣款或转账的家庭。 |
| `targetCount` | 群体任务目标人数。 |
| `adjustmentId` | 传令时选择的经济选项。 |
| `amount` | 本次成本、赏钱或扣罚金额。 |

失败处置事件字段：

| 字段 | 含义 |
|---|---|
| `punishmentId` / `punishmentLabel` | 处置类型，如宽恕、训斥、加派补做、扣月银。 |
| `triggerType` | 触发方式，如首次失败、连续失败、本月累计、严重失败、手动复核。 |
| `requiresApproval` | 是否需要管账人/上级背书。 |
| `executorId` | 建议执行钱项处罚的人。 |
| `moneyPenalty` | 钱项处罚上下文，含金额、来源、权限说明。 |

`order:impact` 只在管理任务产生明确结局时记录：

- 完成：`status = completed`
- 失败：`status = failed`
- 拒绝：`status = declined`
- 超时：`status = expired`

当前 `orderDelta` 是调试估算，不是正式秩序结算。正式结算系统上线前，所有秩序参数只作为解释和调参依据。

## 4.2 常用查询

```js
// 看所有管理任务事件
BehaviorTelemetry.query({ event: 'order:impact', limit: 200 })

// 看黛玉相关任务和人物影响
BehaviorTelemetry.query({ charId: 'daiyu', limit: 500 })
  .filter(r => r.event.startsWith('quest:') || r.event === 'character:effect')

// 看任务造成的技能变化
BehaviorTelemetry.query({ event: 'character:effect', limit: 500 })
  .filter(r => r.effectType === 'skillXp')

// 汇总某个人：会包含 managementQuests / characterEffects / orderImpacts
BehaviorTelemetry.summarize('daiyu')

// 看任务经济压力
BehaviorTelemetry.query({ limit: 500 })
  .filter(r => ['economy:quest_cost', 'economy:quest_reward', 'economy:quest_fine'].includes(r.event))

// 看失败后怎么处置
BehaviorTelemetry.query({ limit: 500 })
  .filter(r => r.event.startsWith('punishment:'))
```
