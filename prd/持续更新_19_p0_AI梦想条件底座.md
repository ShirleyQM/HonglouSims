# AI 与梦想系统 P0：条件底座 PRD

> 日期：2026-06-23
> 父文档：`持续更新_19_AI与梦想系统联动.md`
> P0 目标：先补齐梦想达成条件能被统一判定、记录和调试的底座，不急着把所有梦想玩法做满。

## 1. P0 范围

本阶段只做四个基础能力：

| 能力 | 目的 | 首版代码入口 |
|---|---|---|
| `DreamConditionRegistry` | 把梦想条件从散乱字段变成统一可判定项 | `js/systems/dream-condition.js` |
| `DreamProgressStore` | 记录每个人梦想的计数、里程碑、最近推进和失败原因 | `js/systems/dream-progress.js` |
| `BehaviorDailyStats` | 统计每日最低需求、行为标签、访问场景、连续达标天数 | `js/systems/behavior-daily-stats.js` |
| `HealthSystem` | 给人物增加健康度、生病、重病、生病天数和照顾事件 | `js/systems/health.js` |

P0 不做：

- 不做完整背包、秘密、证据、声望分域。
- 不做所有梦想的完整结局。
- 不强制改变 AI 行为；只允许 Dream/Intent 后续读取这些结果。
- 不让梦想绕过基础需求、礼法、场景权限、任务限制。

## 2. 数据口径

## 3. HealthSystem

### 3.1 健康度

`health` 是 0 到 100 的长期身体状态，不等于精力：

| 区间 | 语义 | 行为影响 |
|---|---|---|
| 80-100 | 康健 | 无额外限制 |
| 60-79 | 微弱 | 更容易疲惫，生病风险轻微提高 |
| 40-59 | 欠安 | 休息/吃饭/照顾权重提高 |
| 20-39 | 病弱 | 可触发 `病中`，部分行动降权 |
| 0-19 | 危急 | 强制优先休养/照顾，梦想风险升高 |

### 3.2 健康变化来源

| 来源 | 影响 |
|---|---|
| 饥饿过低 | 每日结算扣健康 |
| 精力过低 | 每日结算扣健康 |
| 心绪长期过低 | 小幅扣健康 |
| 生病状态持续 | 持续扣健康 |
| 睡眠、饮食、照顾、药汤 | 恢复健康 |
| 体弱专属性格 | 降低健康恢复、提高生病概率 |

### 3.3 P0 API

```js
HealthSystem.initChar(char)
HealthSystem.getHealth(charId)
HealthSystem.changeHealth(charId, delta, reason)
HealthSystem.isSick(charId)
HealthSystem.getIllnessLevel(charId) // none/light/sick/severe/critical
HealthSystem.getSickDays(charId, windowDays)
HealthSystem.recordCareEvent(actorId, targetId, reason)
```

## 4. BehaviorDailyStats

### 4.1 统计内容

| 指标 | 来源 | 用途 |
|---|---|---|
| `needMin` | 每日 tick 汇总 | `hungerMin`、`energyMin`、`moodMin` |
| `actionTagMinutes` | 行动开始/完成事件 | 独处日、娱乐日、守护日 |
| `visitedSceneIds` | 场景进入/移动事件 | `visitedScenesMin` |
| `worldEventTags` | 大事件事件流 | `disasterFreeDays` |
| `flags` | 日结计算 | `funSatisfiedDay`、`solitudeDay`、`healthSafeDay` |

### 4.2 P0 API

```js

```

## 5. DreamProgressStore

### 5.1 计数规则

所有梦想计数统一进入 `dreamProgress.counters`：

```js
DreamProgressStore.addCounter(charId, key, delta, source)
DreamProgressStore.setCounter(charId, key, value, source)
DreamProgressStore.getCounter(charId, key)
DreamProgressStore.markMilestone(charId, key, value, source)
DreamProgressStore.recordReason(charId, dreamType, reason)
```

事件来源可以是：

- 任务完成：`quest:completed`
- 家具完成：`furniture:complete`
- 互动完成：`interaction:completed`
- 健康变化：`health:changed`
- 关系变化：`relation:change`
- 手动调试：后台按钮

### 5.2 P0 允许先手动推进

背包、秘密、证据等 P1 才会实体化。P0 对这些条件先允许通过后台或事件写入计数，例如：

```js
DreamProgressStore.addCounter('yinger', 'craftedWorks', 1, { source: 'debug' })
DreamProgressStore.addCounter('xueyan', 'discoveredSecrets', 1, { source: 'debug' })
```

## 6. DreamConditionRegistry

### 6.1 判定结果

每个条件返回统一结构：

```js
{
  key: 'healthMin',
  label: '健康不低于 60',
  ok: true,
  value: 72,
  target: 60,
  progress: 1,
  source: 'HealthSystem',
  reason: '当前健康 72'
}
```

梦想总结果：

```js
{
  charId,
  dreamType,
  ok,
  progressScore,
  unmet: [],
  met: [],
  unknown: []
}
```

### 6.2 P0 API

```js
DreamConditionRegistry.normalizeConditions(rawConditions)
DreamConditionRegistry.evaluateCondition(char, condition, context)
DreamConditionRegistry.evaluateDream(charId, dreamType)
DreamConditionRegistry.describeCondition(condition)
```

## 7. 后台 v2 P0

梦想系统页先增加一个“P0 条件诊断”区：

| 区块 | 内容 |
|---|---|
| 当前人物 | 梦想类型、标题、当前健康、所属家庭、公账 |
| 条件列表 | 每条条件的当前值、目标值、是否满足、数据来源 |
| 缺口列表 | 当前无法真实判定、只能计数或未知的条件 |
| 调试推进 | 对 P0 计数器执行 +1/-1/清零 |
| 日统计 | 最近 7 日最低需求、访问场景数、娱乐日、独处日 |

后台不展示大 JSON 作为主入口；JSON 只保留在高级折叠区。

## 8. 验收标准

| 验收项 | 标准 |
|---|---|
| 健康度 | 每个人初始化 `health`，低需求会影响健康，健康变化有事件。 |
| 生病 | 至少能通过状态/健康阈值判断 `isSick` 与 `sickDays`。 |
| 日统计 | 连续 3 日模拟后，能看到每人每日最低需求和访问场景。 |
| 梦想计数 | 能通过任务/调试推进 `craftedWorks`、`discoveredSecrets` 等计数。 |
| 条件判定 | 至少覆盖 `familyFundMin`、`healthMin`、`needMin`、`relationAxisMin`、`counterMin`、`visitedScenesMin`。 |
| 后台诊断 | 选择人物后能看到梦想条件满足/未满足/未知三类结果。 |
| 不破坏现有 AI | 连续模拟不因 P0 新系统报错；无梦想数据的人保持原行为。 |

## 9. 实施顺序

1. 新增 `HealthSystem`，只接初始化、日结、事件。
2. 新增 `BehaviorDailyStats`，先统计需求最低值和访问场景。
3. 新增 `DreamProgressStore`，提供计数和里程碑。
4. 新增 `DreamConditionRegistry`，先覆盖 P0 条件。
5. 后台 v2 梦想页增加条件诊断和调试推进。
6. 跑 3 日模拟与后台连通性检查。
