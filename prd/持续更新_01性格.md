## 人物专属性格（specialtyMetadata）

人物专属性格用于承载“这个人独有的好玩行为”，例如宝玉“见女儿痴语”、黛玉“临风洒泪”、凤姐“辣语压人”。它和通用性格不同：

- 通用性格 `traitMetadata` 是可复用维度，例如“热络 / 孤僻”“爱洁 / 邋遢”，用于框架型 AI 差异。
- 人物专属性格 `specialtyMetadata` 是角色个体化梗、执念、病症、日常职责或旁观反应，不与通用性格互斥。
- 旧的 `profiles[charId].specialties` 仍是“某人物拥有哪几个专属性格”的分配表；新的 `specialtyMetadata` 是“每个专属性格具体是什么、影响什么系统、首版参数是多少”的定义表。
- 如果不同人物复用同一个 specialtyId，例如 `mediate`，全局定义 key 使用 `人物ID.specialtyId`，例如 `baochai.mediate`、`sheyue.mediate`，避免含义串线。

首版数据协议：

```js
charSpecialtyConfig: {
  specialtyMetadata: {
    female_charm: {
      ownerId: 'baoyu',
      label: '见女儿痴语',
      category: '专属性格',
      description: '宝玉遇到女性社交对象时更容易主动开口，但只轻微影响女性社交大类，不能盖过关系锚点。',
      systems: ['AI权重'],
      effectExamples: ['女性社交大类权重 x1.12，上限 x1.15', '争执类互动不加权'],
      effects: {
        actionWeights: { female_social: 1.12 },
        social: { femaleCategoryWeight: 1.12, maxMultiplier: 1.15, relationAnchorFirst: true }
      }
    }
  }
}
```

## 性格对偶关系（持续维护）

性格的成对关系维护在 `js/config.js` 的 `charSpecialtyConfig.traitMetadata` 中，字段为：

```js
oppositeTrait: 'beiguan'
```

约定：

- `oppositeTrait` 是现有性格之间的配置关系，不新增性格、不新增效果类型。
- 对偶双方需要互相回指，例如 `leitian.oppositeTrait = 'beiguan'`，同时 `beiguan.oppositeTrait = 'leitian'`。
- 后台【人物 / 性格系统】按对偶分组展示；在表格里改“对偶”列会自动维护双方回指。
- 真正效果仍维护在各性格已有的 `effects` 内，由 `TraitEffectSystem` 和现有 AI / 需求 / 状态 / 关系 / 任务链路读取。
- 当前已有性格中，`shishui`（嗜睡）没有现成对偶；按“不引入新内容”的原则暂列为未成对。

## 一、当前已经有什么

当前性格的主链路是：

```text
人物 profiles.aiTraits
  -> getCharTraits(c)
  -> CharSpecialtySystem.traitModTable()
  -> AI 候选行动的 tags
  -> calcTraitFactor()
  -> finalWeight
  -> NPC 更可能选择某类行动
```

相关代码：

- `js/config.js`：`charSpecialtyConfig.traitMetadata`、`traitModifiers`、`profiles`
- `js/ai.js`：`calcTraitFactor()`、`finalizeCandidate()`
- `js/systems/specialty.js`：性格权重表、人物专长和需求修正
- `js/systems/character-effects.js`：需求、状态、关系、记忆等离散变化的统一入口

目前真正能产生行为差异的主要是 `traitModifiers.boost/cut`。它只能回答：

> “这个 NPC 更想做哪类行动？”

它还不能统一回答：

- 情绪和状态多容易触发、持续多久、恢复多快
- 一次互动带来的关系变化应该放大还是缩小
- NPC 是否记住事件、记多久、是否记仇
- NPC 是否接受任务、是否准时、是否拖到截止前
- 需求的临界点和衰减速度是否因性格不同
- 花钱、争执、群聊、认输等行为概率是否变化

因此 Metadata 中的说明和效果示例目前大部分只是说明文字，不会自动执行。

## 二、还缺的核心：统一性格效果系统

建议新增：

```text
js/systems/trait-effects.js
```

系统职责是读取人物全部 `aiTraits`，把多个性格合成为一组运行时修正，不直接硬编码具体人物。

### 1. 建议的 Metadata 数据协议

每个性格增加结构化 `effects`，文字说明继续保留给后台展示：

```js
mingan: {
  label: '敏感',
  category: '性情',
  description: '心思细腻，容易受伤。',
  effectExamples: ['关系值变化效果增强', '更易触发尴尬'],
  effects: {
    relation: {
      positiveMultiplier: 1.35,
      negativeMultiplier: 1.8
    },
    stateChance: {
      awkward: 1.5,
      melancholy: 1.25
    },
    memoryChanceMultiplier: 1.3
  }
}
```

`effects` 建议支持以下模块：

| 模块 | 用途 |
|---|---|
| `actionWeights` | 行动标签权重，逐步替代单独的 `traitModifiers` |
| `needCoeffs` | 需求衰减、恢复、上下限修正 |
| `needThresholds` | 告急、触发状态、开始满足需求的阈值 |
| `stateChance` | 状态触发概率 |
| `stateDuration` | 状态持续时间倍率 |
| `stateRecovery` | 状态恢复速度倍率 |
| `relation` | 好感、信任、友谊变化倍率 |
| `memory` | 记忆生成率、负面记忆保持、遗忘速度 |
| `social` | 主动社交、群聊加入、冲突升级、认输等概率 |
| `quest` | 接受率、准时率、截止前加权、失败概率 |
| `schedule` | 起床、睡眠、日程容差 |
| `movement` | 移动速度或漫游倾向 |
| `money` | 消费、积蓄、施舍等倾向 |

### 2. 建议提供的统一接口

```js
TraitEffectSystem.has(c, traitId)
TraitEffectSystem.get(c, path, fallback)
TraitEffectSystem.multiply(c, path, base)
TraitEffectSystem.modifyActionWeight(c, cand, factor)
TraitEffectSystem.modifyNeedCoeffs(c, coeffs)
TraitEffectSystem.modifyNeedThreshold(c, needKey, threshold)
TraitEffectSystem.modifyStateChance(c, stateId, probability)
TraitEffectSystem.modifyStateDuration(c, stateId, duration)
TraitEffectSystem.modifyRelationDelta(actor, target, axis, delta, context)
TraitEffectSystem.modifyMemoryChance(c, memory, probability)
TraitEffectSystem.modifyQuestDecision(c, quest, decision)
TraitEffectSystem.modifySchedule(c, slot)
```

多个性格组合时建议：

- 倍率相乘，但最终统一限制在合理区间，例如 `0.2~3`
- 概率先加权再限制在 `0~1`
- 正负关系变化分开计算
- 互相冲突的性格先允许共存，但后台给出诊断提示
- 所有修正写入 AI 决策日志，便于解释“为什么 NPC 这么做”

## 三、需要接入的现有系统

### P0：必须接入

1. `js/ai.js`
   - 保留现有行动权重入口。
   - 增加群体密度、等待、室内外、任务紧迫度等可识别标签。

2. `js/action-queue.js`
   - `calcNeedCoeffs()` 接入性格需求修正。
   - `checkNeedTriggers()` 接入个体化阈值。
   - 移动计算接入性格速度倍率。

3. `js/systems/character-effects.js`
   - 关系、状态、记忆统一在这里应用性格修正。
   - 必须避免调用方各自重复修正。

4. `js/systems/relation.js`
   - `changeAxis()` 支持传入 actor、target、原因。
   - 当前关系是共享轴，需要明确“谁受到影响”。例如敏感应放大敏感者受到的体验，而不是无条件双向放大。

5. `js/systems/quest.js`
   - 接任务概率、任务候选权重、截止前行为和日程准时率接入性格。

### P1：增强表现

6. `js/systems/multi-interact.js`
   - 情绪传染、旁观反应、群聊参与接入性格。

7. `js/systems/dialogue.js`
   - 当前群聊模块仍优先读取旧 `CHAR_DEFAULT_TRAITS`，应改为统一 `getCharTraits()`。
   - 对话轮次、加入概率、直言/圆滑风格接入性格。

8. `js/systems/ai-drama.js`
   - 冲突、和解、求爱、安慰等意图概率接入性格。

9. `js/systems/narrative.js`
   - 气泡只负责表现，不负责真正效果。
   - 可根据性格选择不同台词风格。

## 六、实现顺序

### 第一阶段：让所有性格至少改变行动

1. 为 AI 候选补齐通用标签：
   - `social`、`conflict`、`waiting`、`indoor`、`outdoor`
   - `meal`、`sleep`、`clean`
   - `task`、`taskUrgent`
2. 为现有性格填写 `actionWeights`。
3. 后台显示“已接入效果”和“仅说明未接入效果”。

完成标准：同一场景模拟一天，不同性格的行动分布有明显差异。

### 第二阶段：需求、情绪、关系

1. 接入 `needCoeffs` 与 `needThresholds`。
2. 在 `CharacterEffectSystem` 接入状态概率、持续时间。
3. 关系变化增加 actor/recipient/context，再接入性格倍率。

完成标准：

- 敏感人物对冲突反应明显更强
- 温和人物更少进入愤怒
- 嗜睡、贪嘴、爱洁人物会更早处理对应需求

### 第三阶段：记忆、群聊、任务

1. 记忆增加 `strength`、`createdAt`、`expiresAt` 或衰减值。
2. 群聊系统改用统一 `getCharTraits()`。
3. 守时、拖延影响任务准备与截止前权重。
4. 多疑、轻信影响任务、邀请和社交接受率。

完成标准：

- 迟钝人物比敏感人物产生更少长期记忆
- 热络人物更常加入群聊，孤僻人物更常离开人群
- 守时人物提前行动，拖延人物临近截止才行动

### 第四阶段：消费、竞赛与长期统计

1. 增加自主消费/施舍候选，支持豪爽。
2. 增加竞赛结果事件，支持好胜和谦让。
3. 记录每个性格的行为统计，用于数值调优。

## 七、调试与可解释性

每次 AI 决策建议保存：

```js
{
  charId: 'daiyu',
  action: '独坐书案',
  baseWeight: 1,
  traitEffects: [
    { trait: 'gupi', key: 'solitude', multiplier: 1.7 },
    { trait: 'xijing', key: 'desk', multiplier: 1.35 }
  ],
  finalWeight: 2.295
}
```

配置后台人物总览增加：

- 最近 10 次决策及性格倍率
- 当前需求阈值
- 当前状态触发/恢复倍率
- 当前关系变化倍率
- 未识别性格效果诊断

这样可以区分：

- 性格确实生效但随机没选中
- 行动缺少正确标签
- Metadata 写了效果但运行时尚不支持
- 多个性格互相抵消

## 八、测试建议

### 单元级

- 每个性格至少测试一个关键效果。
- 多性格倍率组合后必须限制范围。
- 删除未知性格时不能令 AI 报错。
- 旧存档没有 `traitMetadata.effects` 时保持原行为。

### 模拟级

固定随机种子，运行每个测试人物 3 个游戏日，统计：

- 各行动类别次数
- 情绪状态触发次数与平均持续时间
- 主动社交次数、群聊加入率、冲突次数
- 需求告急次数
- 任务准时率与失败率
- 正负记忆数量

上述模拟数据现由 `BehaviorTelemetry` 采集。默认观察黛玉，可用
`BehaviorTelemetry.watch(charId, true)` 增加观察角色，并用
`BehaviorTelemetry.download(charId)` 导出 JSON。每次决策会保留权重前六名和各乘数，
可直接区分“没有社交候选”“社交被候选池截断”“社交存在但权重落选”。

新增的交游与心绪需求通过 `needDriveFactor` 进入最终行为权重：

- 孤僻、喜静等性格决定角色平常是否偏爱独处。
- 交游低于 40 后，基础需求会逐步压过独处偏好，防止角色永久闭门不出。
- 心绪低时，根据性格在慰藉、倾诉、独处、大怒、犯痴或写诗之间分化。
- 性格仍决定表现方式，需求只提供必须处理的长期压力。

建议判定：

- 性格对应行为较无性格基线至少变化 20%
- 不应出现单个倍率令 NPC 永远只做一种行动
- 生存需求告急时，性格不能覆盖紧急需求处理

## 九、结论

当前缺的不是继续增加标签，而是增加一个统一的“性格效果解释层”。

最稳妥的结构是：

```text
traitMetadata.effects（配置）
  -> TraitEffectSystem（统一解释与合并）
  -> AI / 需求 / 状态 / 关系 / 记忆 / 任务（各系统调用）
  -> 决策日志与后台诊断（验证）
```

建议先完成第一、二阶段。这样现有性格都会在日常行为、需求节奏和情绪关系上产生可见差异，再继续做记忆、任务和长期追求。
