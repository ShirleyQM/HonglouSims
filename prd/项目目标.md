# 大观园模拟项目目标

> 更新时间：2026-06-25
> 文档定位：用于确定项目阶段目标、发版边界和后续取舍。

## 1. 一句话目标

《大观园模拟》是一款以红楼大观园为舞台的多人物日常模拟游戏：玩家观察和轻度干预人物的一天，人物会根据需求、性格、关系、职业职责、起居和任务自主行动，并产生可解释的社交、状态和故事变化。

## 2. 当前版本定位

当前版本定位为 `v0.1-alpha`，目标是“稳定好玩”的公开试玩原型：

- 给朋友和早期玩家试玩的静态网页原型。
- 重点验证“玩家能否理解并完成一段大观园生活目标”。
- 核心乐趣优先级：养成 + 管理，其次是观察和轻度捣乱。
- 先做单机本地存档，不做账号、云端数据库和多人在线。
- 后台只服务开发者和策划，不对普通玩家开放配置编辑。

## 4. P0：下一版 playable 目标

目标：让外部玩家打开页面后能玩 15-30 分钟，完成一个明确目标，并理解移动、社交、管理、日常四类基础操作。

P0 新手目标：

```text
林黛玉进贾府
  -> 移动：随引导进入指定地点或靠近关键人物
  -> 社交：与贾母、宝玉或迎接人物完成一次互动
  -> 管理：通过传令安排随侍/丫鬟完成一件小事
  -> 日常：安排或观察一次起居/需求相关活动
  -> 目标完成：玩家看到明确反馈，理解黛玉已安顿进府
```

### P0 必须稳住

- 页面打开无致命 JS 错误。
- 人物不会卡进空白地图或不可达区域。
- 点击地面、家具、人物、传令、起居、人物面板都能理解。
- 起居对实际 AI 有可见影响，需求活动能找到真实家具/社交候选。
- 传令能展示可命令对象、服从/信任/接受概率，并生成任务。
- 存档按钮有明确反馈，读档/清档不误导。
- 后台入口不影响普通玩家试玩。
- 新手引导步骤可以完成、可回看目标、失败时有提示。

### P0 可暂缓

- 云端账号和数据库。
- 完整编辑器和玩家自定义配置。
- 大规模剧情结局。
- 多地图完整美术化。
- 全自动深度多日模拟验收。

## 5. P1：系统可扩展目标

目标：让新增人物、职业、任务、互动、家具、状态不需要大改代码。

重点：

- 职业系统与故事系统拆清楚：职业负责身份、职责、日课和管理权限；故事系统负责路径、节点和长期剧情。
- 互动模板、家具动作、任务模板继续配置化，减少写死逻辑。
- `SocialContextSystem` 或等价解释层统一身份、名分、关系、礼法和传令权限。
- 后台所有策划常用页保持表格式，少用大段 JSON。
- 测试矩阵从冒烟测试逐步扩展到场景自动化。

## 6. P2：长期游戏目标

目标：从“好玩的模拟玩具”走向“可长期观察和干预的红楼群像沙盒”。

可能方向：

- 多日事件和人物命运线。
- 梦想、秘密、声望、职业晋升形成长期叙事压力。
- 更多可配置职业组织：府内管事、丫鬟、小厮、先生、外客、商人、官场。
- 玩家能创建或调配置，但需要数据库/云同步后再考虑开放。
- 更完整的地图、美术资产和角色立绘。

## 7. 当前关键设计原则

- 前台是玩家体验，不展示后台 JSON 和调试细节。
- 后台是可配置总览，尽量表格化，隐藏暂时不用的配置页。
- 所有人物行为尽量走同一底层：需求、起居、职业、任务、家具、社交都进入 Utility AI 和行动队列。
- 职业日课进入起居，临时传令不进入起居。
- 旧日程和基础作息只做兜底，新起居覆盖其上。
- 关系里的服从、信任必须影响传令和任务接受。
- 玩家不需要懂系统名，也能看懂“此人为什么这么做”。
- 新手引导必须用真实系统完成，不做假按钮和独立脚本演出。

- 发版测试增加新手链路冒烟。
