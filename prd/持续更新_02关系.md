# 持续更新 02：关系系统

> 本文档是关系系统的持续维护 PRD。当前实现以“好感 / 信任 / 友谊 / 服从”四轴为核心，新后台【关系】-【关系标签】负责配置显示名、阶段标签和摘要标签。

## 2026-06-23 置顶补充：与身份礼法的统一口径

关系系统不再单独承担“古代礼法”的全部语义。后续统一为上层 `SocialContextSystem` 汇总解释：

- 身份系统回答：这个人自身是什么位阶。
- 关系系统回答：这两个人名义上是什么关系、实际处得怎么样。
- 礼法系统回答：在当前场景和行为下，这件事合不合规矩、有没有风险。
- 声望系统回答：这件事被谁认可或厌恶，影响哪个圈层的名声。

### A. 关系系统在统一模型中的职责

| 职责 | 当前来源 | 未来输出给 `SocialContext` | 说明 |
|---|---|---|---|
| 名分关系 | `relationInit.initType` | `nominalRelation` | 父子、母子、祖孙、夫妻、主仆、朋友、同事、亲戚等；这是两人之间的社会名分。 |
| 亲缘类型 | `getKinshipType(initType)` | `kinshipType` | 血缘、姻亲、无血缘等；用于称谓、孝道、禁忌和综合关系标签。 |
| 实际关系四轴 | `relations` | `affection / trust / friendship / submission` | 好感、信任、友谊为双向共享；服从为方向性。 |
| 综合关系分 | `REL_AXIS_WEIGHTS` | `relationScore` | 只描述实际亲疏，不等同于身份高低或礼法许可。 |
| 综合关系标签 | `relationPanelConfig.compositeRelationRules` | `compositeRelationLabel` | 例如父慈子孝、心腹忠仆、莫逆之交；由名分关系 + 关系轴共同命中。 |
| 关系事件 | 待补齐 | `relationEvents` | 记录谁对谁、因何事、哪条轴变化，用于记忆、AI、声望和剧情。 |

### B. 与身份系统的边界

身份位阶和两人关系必须拆开：

| 概念 | 示例 | 属于谁 | 不应混淆为 |
|---|---|---|---|
| 身份位阶 | 至尊、长辈、平辈主子、晚辈主子、体面仆从、普通仆从、外人 | 身份系统 | 家庭关系、好感阶段 |
| 名分关系 | 父子、母子、祖孙、夫妻、主仆、朋友、亲戚 | 关系系统 | 社会位阶 |
| 实际关系 | 亲近、疏远、信任、猜疑、服从、怨怼 | 关系系统 | 名义身份 |
| 礼法判定 | allowed / conditional / risky / forbidden | 礼法系统 | 关系标签 |

例如：

- “宝玉是晚辈主子”是身份位阶。
- “宝玉和袭人是主仆”是名分关系。
- “袭人对宝玉服从 70、信任 60”是实际关系。
- “宝玉夜里单独召袭人进房是否逾矩”是礼法判定，需要身份、名分、实际关系、场景、见证者一起判断。

### C. 与礼法规则的关系

关系系统只提供判断材料，不直接决定礼法结果：

- `relationScore`、`trust`、`affection` 可以影响 `conditional` 行为是否放行。
- `submission` 可以影响主仆任务接受、差遣优先级、推拒概率。
- `compositeRelationLabel` 可以用于叙事和提示，但不应直接作为硬权限。
- 礼法系统最终仍以 `SocialContextSystem.evaluateProtocol()` 输出为准。

也就是说，“关系好”可以降低风险或提高成功率，但不能自动抹掉身份越界；“关系差”可以增加拒绝、告密、怨气，但不自动改变两人的名分。

### D. 与声望系统的关系

声望应读取关系事件和礼法事件，但不替代关系分。

建议关系事件携带：

```js
{
  actorId,
  targetId,
  source: "interaction / quest / scene / story",
  axisDeltas: { affection: 5, trust: -3, submission: 0 },
  socialContext: {
    actorRank,
    targetRank,
    nominalRelation,
    kinshipType,
    protocolBehavior
  },
  reputationImpact: {
    household: 0,
    masterCircle: 0,
    servantCircle: 0,
    clan: 0,
    outside: 0
  }
}
```

主子与仆从的声望解释不同：

- 主子因宽厚、守礼、能管家获得体面；滥罚、偏宠、失礼会损害家内与长辈圈评价。
- 仆从因忠心、能干、嘴严、得主子信任获得声望；搬弄是非、越礼、失职会影响主子圈或仆从圈。
- 同一事件可以同时改变关系和声望，例如“替主子遮掩”提高主仆信任，也可能提高仆从圈威信或带来外部风险。

### E. 后续改造顺序

| 阶段 | 目标 | 验收 |
|---|---|---|
| Phase 1 | 关系系统暴露标准化 `RelationFacet`，供 `SocialContextSystem.resolve()` 读取。 | 任意两人可输出名分关系、亲缘类型、四轴、综合分、综合关系标签。 |
| Phase 2 | 所有关系变化统一走事件 payload。 | 日志能解释“谁因为什么对谁的哪条轴变化”。 |
| Phase 3 | AI、任务、互动门槛改读 `SocialContext`，不再各自猜身份和关系。 | 主仆差遣、亲友靠近、长幼避嫌等行为解释一致。 |
| Phase 4 | 关系阈值事件接入记忆和叙事气泡。 | 关系跨阶段时产生可见反馈和可追踪记忆。 |
| Phase 5 | 声望读取关系事件和礼法事件。 | 同一事件可以分别影响关系、主子圈声望、仆从圈声望和家内名声。 |

## 2026-06-21 置顶补充：关系驱动下一阶段

当前关系系统已经能记录和展示四轴关系，但下一阶段要从“面板数值”升级为“行为驱动器”：关系不仅影响互动结果，还要反过来影响 NPC 想靠近谁、回避谁、照顾谁、差遣谁、服从谁、向谁求助、因谁吃醋或记恨。

### 核心目标

- 关系进入 AI 候选：关系分、关系类型、近期变化和身份方向都应影响社交、跟随、邀请、争执、慰藉、任务下发。
- 关系产生记忆：重大正负变化、阈值跨越、被拒绝、被帮助、被撞见等要形成可追踪记忆。
- 关系影响权限：主仆、长幼、亲疏、信任和服从共同决定能否进入私密场景、能否下发任务、是否会被拦。
- 关系触发事件：跨阈值不只显示标签，还能触发气泡、状态、旁观反应、流言或小剧情节点。
- 关系可解释：每次关系变化要能说明来源、影响轴、受影响者和上下文。

### 优先需求

| 优先级 | 需求 | 说明 | 验收方式 |
|---|---|---|---|
| P0 | 关系上下文标准化 | 关系变化统一携带 `actor / recipient / axis / delta / source / context`。 | 日志和埋点能区分“谁对谁的感受变了”。 |
| P0 | 关系驱动社交候选 | AI 生成社交候选时读取综合分、单轴、近期互动和身份关系。 | 高好感/高信任人物更常主动靠近；低信任更常回避或观察。 |
| P1 | 阈值事件落地 | 关系跨阶段时触发 `relationThresholdEvents`：气泡、状态、记忆、旁观反应。 | 关系从冷淡到亲近、从信任到猜疑时有可见反馈。 |
| P1 | 服从/体恤行为化 | 仆从对主子服从影响任务接受、任务优先级、拖延和怨气；主子体恤影响惩罚/赏赐/慰藉。 | 主仆关系能产生明显不同的日常行为。 |
| P1 | 关系记忆与衰减 | 重大互动写入记忆，并随时间衰减或被重复事件强化。 | 模拟报告能看到“因某事亲近/疏远”的链路。 |
| P2 | 三角关系/旁观反应 | A 与 B 关系变化可影响 C 的嫉妒、护短、站队、流言。 | 群体场景中出现自然的旁观反馈。 |

### 设计原则

1. 好感、信任、友谊继续作为双向共享轴；服从继续保留方向性。
2. 新增“关系上下文”不改变旧数值结构，先从事件 payload 和 CharacterEffectSystem 入口补齐。
3. 关系不直接替代 Utility AI，而是作为候选来源和乘法因子影响选择。
4. 关系阈值事件必须可配置、可埋点、可在长模拟报告中统计。
5. 所有关系驱动行为都要尊重身份礼法和场景权限。

## 2. 数据模型

### 2.1 关系存储

运行时关系存在全局 `relations`，key 为排序后的 `idA|idB`。

```js
{
  affection: 20,
  trust: 16,
  friendship: 10,
  submission: { baoyu: 0, xiren: 40 },
  initType: "主仆",
  note: "",
  lastInteract: 12345,
  memoryIds: [],
  thresholdEvents: []
}
```

### 2.2 轴定义

| 轴 | ID | 范围 | 方向 | 当前用途 |
|---|---|---:|---|---|
| 好感 / 姻缘 | `affection` | -100~100 | 双向共享 | 情感倾向，变化快；关系图右上 |
| 信任 | `trust` | -100~100 | 双向共享 | 可靠感，积累慢、崩塌快；关系图左下 |
| 友谊 | `friendship` | -100~100 | 双向共享 | 共同经历，极慢衰减；关系图左上 |
| 服从 | `submission` | -100~100 | 方向性 | A 对 B 的顺从程度；仆从看主子时显示右下 |
| 体恤 | `care` | -100~100 | 显示派生 | 主子看仆从时右下显示；当前取 `affection` 数值但用独立阶段标签 |

### 2.3 综合分

```js
score = affection * 0.45 + trust * 0.35 + friendship * 0.20
```

规则：
- `submission` 不进入综合分。
- 五个显示轴统一为 -100~100，阶段标签直接按真实值匹配。
- 关系面板外环进度条必须按真实轴范围绘制：起始边为 -100，扇形中点为 0，终止边为 +100；负值逆时针、正值顺时针。

阶段表展示方式：

- 孝道和慈爱作为一组 pair 展示。

- 标签名允许为空，保存时仍保留该区间。
- 表现说明列融合 Excel 的“描述”；“可用互动”作为示例单独展示，不代表当前已经解锁互动。
- 数值区间底层统一左闭右开：`[开端, 下一行开端)`；后台展示为策划易读的 `-50 ~ -21`。
- 后台只允许配置每行“开端”数字；右边界由下一行开端自动生成，最后一行右边界默认为 100。
- 保存时校验：第一行开端必须为 -100；开端必须是 -100~100 的数字；各行开端必须严格递增；否则不保存。

### 3.5 综合关系模块

综合关系不是单纯综合分阈值，而是“家庭关系 + 关系轴条件”的组合标签。

`import systems/0619新增综合关系.xlsx` 已导入默认配置，共 39 条规则。

规则字段：

| 字段 | 说明 |
|---|---|
| 家庭关系 | 父子、母子、祖孙、夫妻、手足、主仆、寄居、无血缘等 |
| 友谊 / 姻缘 / 信任 | 支持 `≥40`、`≤-20`、`—` |
| 第四象限 | 支持 `孝道≥60`、`慈爱≥80`、`服从≥60+恩恤≥70`、`情分≥40`、`意气≥30` 等 |
| 综合关系标签 | 命中后显示在关系圆图中心，例如“父慈子孝”“心腹忠仆”“莫逆之交” |
| 示例 | 策划参考，不参与运行匹配 |

当前运行映射：
- `服从` / `孝道`：取双方 `submission` 中较高的一侧。
- `恩恤` / `慈爱` / `情分`：暂取 `affection`。
- `意气`：暂取 `friendship`。
- 规则按表格顺序匹配，第一条命中即返回综合关系标签。

| 右下 | 服从 / 体恤 / 孝道 / 慈爱 / 空 | 主仆显示服从/体恤；父母子女/祖孙显示孝道/慈爱；其他关系先留空 |

进度条规则：
- 每个扇形外边缘有一条弧形进度条。
- 起始边代表 -100，终止边代表 +100。
- 0 值在扇形弧度中点。
- 正值顺时针填充，负值逆时针填充。
- 名字下方不重复展示“友谊 / 姻缘 / 信任 / 第四象限”的文字摘要，具体标签已经在四象限图内展示。
- 卡片右上角的 `+N/-N` 是综合关系分。
- 圆心文字：在“姻缘、友谊、综合分”三者中取数值最高者对应的【关系标签】；轴标签取 `axisStages.*.label`，综合分标签取 `typeLabel`，不取 `relationName`。
- 圆心综合分：继续显示 `+N/-N`，并用半圆水位表达方向；正值填上半圆，负值填下半圆。后续可扩展成“高山 / 深水”视觉。

### 5.3 LLM 对话中的关系展示

LLM 对话浮层不单独维护一套关系条口径。底层统一读取 `relationAxisDisplayModel(ri, c, t)`：

- 关系网络：把同一套轴模型展示成四象限关系图。
- LLM 对话：把同一套轴模型展示成纵向关系条，顺序为友谊、姻缘、信任、第四象限。
- 对话开始时记录 baseline；每次 `relation:axis_change` / `relation:change` 后刷新。
- 当前对话中上涨的轴显示绿色高亮和 `+N`，下跌的轴显示红色高亮和 `-N`。
- 顶部综合分不再叫“情分”，也不再显示“名分关系 · 初始关系”串；改为和关系图圆心一致的关系标签 + 综合分，例如“萍水相逢 +18”。

## 6. 主仆关系规则

当前实现：
- `submission` 按人物 `socialRank` 差值初始化。
- `getRelationAxis(A, B, 'submission')` 表示 A 对 B 的服从。
- 仆从看主子：右下显示“服从”，阶段来自 `submission`。
- 主子看仆从：右下显示“体恤”，当前数值取 `affection`，阶段来自 `care` 配置。
- 亲缘晚辈看长辈：右下显示“孝道”，当前数值取 `submission`，阶段来自 `filialPiety` 配置。
- 亲缘长辈看晚辈：右下显示“慈爱”，当前数值取 `affection`，阶段来自 `parentalCare` 配置。
- 其他关系：右下留空，暂不展示额外轴。

待实现：
- 贾府丫鬟对主子服从最低不低于 30 的家规下限。
- 王熙凤等特殊主子对服从要求更高。
- 服从过低触发赶走/发卖等剧情。
- 体恤独立数值轴，而不是临时借用好感。

## 7. 血缘与姻亲

当前实现：
- 根据 `relationInit.initType` 推导 `kinshipType`。
- 已支持类型标注：直系血亲、旁系血亲、姻亲、无血缘。
- 尚未进入数值规则。

规划规则：

| 血缘类型 | 示例 | 规则规划 |
|---|---|---|
| 直系血亲 | 父母、子女、祖孙 | 好感/信任基础值 +20，服从受孝道强制影响 |
| 旁系血亲 | 兄弟姐妹、叔伯、姑舅 | 友谊基础值 +15，但可因利益冲突大幅下降 |
| 姻亲 | 嫂子、姐夫、继母 | 好感基础值 +5，服从受家族辈分影响 |
| 无血缘 | 主仆、朋友、外姓亲戚 | 无基础加成，完全靠互动 |

待实现机制：
- 初始关系保底。
- 血缘间关系变化倍率。
- 伤害血亲带来的家族范围惩罚。
- 家族任务对血缘关系的条件检查。
