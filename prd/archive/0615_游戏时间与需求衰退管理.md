# 游戏时间与需求衰退管理

> 更新时间：2026-06-15
> 适用范围：游戏时间、需求自然衰退、家具恢复、状态时长、关系衰退、任务时间及相关存档。
> 主要代码：`js/runtime.js`、`js/action-queue.js`、`js/config.js`、`js/systems/state.js`、`js/systems/relation.js`、`js/systems/quest.js`

## 1. 当前设计目标

- 1 个游戏日对应 20 分钟现实时间。
- 饥饿按一日三餐的节奏下降。
- 精力、娱乐、洁净按一整天的生活节奏下降。
- 人物体质、属性、性格、专长和状态仍能形成明显差异。
- 吃饭、睡觉、游园、梳洗等行为应能覆盖一段合理的需求消耗。
- 洒扫类任务完成后恢复洁净。

## 2. 游戏时间基准

定义位置：`js/runtime.js`

| 项目 | 当前值 | 换算 |
| --- | ---: | --- |
| `GAME_DAY_REAL_MIN` | 20 | 1 游戏日 = 20 现实分钟 |
| `GAME_MINUTES_PER_REAL_SEC` | 1.2 | 1 现实秒 = 1.2 游戏分钟 |
| 1 游戏小时 | 60 游戏分钟 | 50 现实秒 |
| 1 游戏分钟 | - | 约 0.833 现实秒 |
| 初始时间 | 第 1 日 08:00 | `gameDay=1, gameHour=8` |

换算公式：

```text
游戏分钟增量 = 现实秒增量 × GAME_MINUTES_PER_REAL_SEC
             = 现实秒增量 × 1.2
```

主循环位于 `js/main.js`：

```text
每帧：
1. 推进游戏时间
2. 更新状态时长
3. 结算需求自然衰退
4. 更新人物行动和家具恢复
5. 结算关系自然衰退
```

模拟采用 `0.05` 秒固定小步执行，每个渲染帧最多追赶 20 步。活动页面发生卡顿时，未处理的时间保留在 `simBacklog` 中，后续帧继续追赶，不会静默丢失。

页面隐藏时明确暂停模拟并清空积压；重新显示时从当前时刻继续。因此游戏不进行离线推进。

## 3. 时间推进与事件

`tickGameTime(dt)` 将小数游戏分钟累计到 `timeAcc`，每满 1 分钟调用一次 `advanceGameTime(1)`。

`tickGameTime` 会处理当前模拟步内全部完整游戏分钟。卡顿追赶由主循环的固定小步和 `simBacklog` 负责。

| 事件 | 触发时机 | 主要使用者 |
| --- | --- | --- |
| `time:tick` | 每推进 1 游戏分钟 | 任务、叙事扫描、人物专长 |
| `time:hour` | 跨游戏小时 | 故事任务、性格消费行为 |
| `time:day` | 跨游戏日 | 天气、家庭事件、记忆衰减、月银、自动存档 |
| `time:period` | 时段变化 | AI 重算、人生路径 |

当前时段：

| 时段 | 起始小时 |
| --- | ---: |
| 拂晓 `dawn` | 05:00 |
| 上午 `morning` | 08:00 |
| 午后 `noon` | 12:00 |
| 黄昏 `dusk` | 17:00 |
| 夜 `night` | 21:00 |

季节按每 90 个游戏日轮换：春、夏、秋、冬。

### 3.1 经济系统使用的时间点

`js/systems/economy.js` 直接监听游戏时间事件，不使用现实时间计时器：

- `time:hour`：处理贾政、贾琏、贾珍、来旺的上班和下班。
- 下班时：向所属家庭公账发放当天工资。
- `time:day`：检查是否为每 30 日周期的初一，并由王夫人、凤姐发放月银。
- 饭钱不按时间自动扣除，只在角色完成进食家具行为时扣除。

因此加速或调整游戏时间时，经济结算仍跟随游戏时钟；页面隐藏暂停模拟时，工资和月银也不会离线推进。

## 4. 需求基础衰退

配置位置：`DEFAULT_CONFIG.needDecayPerGameDay`，位于 `js/config.js`。

| 需求 | key | 满值人物每日基础下降 | 设计含义 |
| --- | --- | ---: | --- |
| 饥 | `hunger` | 72 | 约每餐消耗 24 |
| 倦 | `energy` | 56 | 一天清醒活动后需要睡眠 |
| 闷 | `fun` | 40 | 一天需要一至两次消遣 |
| 洁 | `hygiene` | 28 | 数日内逐渐需要梳洗 |

最终每帧衰退公式位于 `decayNeeds(c, dt)`：

```text
每游戏分钟基础衰退 = needDecayPerGameDay[需求] ÷ 1440

本帧衰退 =
  每游戏分钟基础衰退
  × 人物 decay
  × 本帧经过的游戏分钟
```

完整写法：

```text
delta =
  dailyDecay[key] / 1440
  × coeff.decay
  × getGameMinuteDelta(dt)
```

需求自然衰退限制在人物对应的 `min` 以上。若人物因性格化家具暂时超过常态
`max`，会从当前值自然回落，不会在下一帧直接夹回常态上限。

当人物正在使用能恢复某项需求的家具时，该项需求暂停自然衰退，避免同一帧一边恢复一边扣除。

## 5. 人物需求系数

每个人物在 `CONFIG.characters[].baseNeedCoeffs` 中定义：

```js
{
  min: 0,
  max: 100,
  grow: 1,
  decay: 1
}
```

| 字段 | 作用 |
| --- | --- |
| `min` | 需求下限 |
| `max` | 需求上限 |
| `grow` | 家具和行为恢复倍率 |
| `decay` | 自然衰退倍率 |

自然衰退使用唯一的 `decay` 倍率。恢复使用 `grow`。

当前 21 名角色的基础 `decay` 范围，不含属性、性格、状态和专长修正：

| 需求 | `decay` 范围 | `grow` 范围 |
| --- | ---: | ---: |
| 饥 | 0.637 - 1.560 | 1.0 |
| 倦 | 0.525 - 1.560 | 0.9 - 1.0 |
| 闷 | 0.455 - 2.100 | 0.8 - 1.1 |
| 洁 | 0.248 - 0.540 | 1.0 |

## 6. 属性修正

配置位置：`DEFAULT_CONFIG.attributeRules`。

属性修正公式：

```text
字段增量 = 配置系数 × (属性值 - 50)
```

| 属性 | 影响 |
| --- | --- |
| 体质 `constitution` | 提高精力上限；降低精力和饥饿衰退 |
| 敏感 `sensitivity` | 提高娱乐的 `decay` |
| 魅力 `charm` | 提高娱乐上限与恢复倍率 |
| 才智 `intellect` | 提高娱乐恢复倍率 |

当前规则：

```js
constitution: {
  energy: { max: 0.008, decay: -0.006 },
  hunger: { decay: -0.004 }
}
sensitivity: { fun: { decay: 0.018 } }
charm: { fun: { max: 0.006, grow: 0.005 } }
intellect: { fun: { grow: 0.004 } }
```

属性修正是在人物基础系数上做加法，之后状态、专长和性格再继续修正。

## 7. 系数叠加顺序

`calcNeedCoeffs(c)` 的顺序如下：

1. 人物 `baseNeedCoeffs`
2. 人物属性 `attributeRules`
3. 当前状态 `stateDefs[].needMods`
4. 临时需求修正 `tempNeedMods`
5. 人物专长 `CharSpecialtySystem.applyNeedMods`
6. 性格特质 `TraitEffectSystem.modifyNeedCoeffs`
7. 长期习惯 `NeedAdaptationSystem.applyCoeffMods`

状态和临时修正通常采用乘法：

```text
coeff[field] = coeff[field] × modifier
```

人物专长使用：

- `decayMult`：乘到 `decay`
- `growMult`：乘到 `grow`

性格 `needCoeffs` 中：

- `decay`、`grow` 使用乘法
- 其他字段使用加法

例：嗜睡 `shishui` 会令精力 `decay × 1.2`。

`needThresholds` 不直接改变衰退速度，只改变 AI 多早开始关注该需求。例如爱洁会更早寻找梳洗行为。

## 8. 家具恢复

恢复公式位于 `updateAction(c, dt)`：

```text
本帧恢复 =
  家具 ratePerGameMin
  × 人物 grow
  × 本帧经过的游戏分钟
```

由于 1 现实秒等于 1.2 游戏分钟，基础总恢复量为：

```text
一次完整使用的基础恢复 =
  ratePerGameMin × duration现实秒 × 1.2
```

字段和后台界面统一使用 `ratePerGameMin`，含义为每游戏分钟恢复量。旧配置中的 `ratePerSec` 会在加载时自动迁移。

| 家具 | 恢复需求 | 使用时间 | 基础总恢复 |
| --- | --- | ---: | ---: |
| 红木书案 | 闷 | 4 秒 | 40.8 |
| 樟木案几 | 倦 | 5 秒 | 12.0 |
| 饭桌 | 饥 | 2 秒 | 28.8 |
| 点心案 | 饥 | 1.5 秒 | 25.2 |
| 厨房灶台 | 饥 | 3 秒 | 36.0 |
| 雕花木床 | 倦 | 7 秒 | 50.4 |
| 竹榻 | 倦 | 3 秒 | 16.2 |
| 浴盆 | 洁 | 5.5 秒 | 57.4 |
| 铜面盆 | 洁 | 2.5 秒 | 30.0 |
| 梳洗妆台 | 洁 | 3.5 秒 | 31.5 |
| 茶案 | 饥 | 2.5 秒 | 33.6 |
| 酒案 | 闷 | 3 秒 | 42.1 |
| 琴台 | 闷 | 4 秒 | 33.6 |
| 花圃 | 闷 | 3 秒 | 31.3 |
| 凉亭 | 倦 / 闷 | 5 秒 | 18.0 / 24.0 |
| 石凳 | 倦 | 2 秒 | 12.0 |
| 轿凳歇脚 | 倦 | 1.5 秒 | 10.8 |

表中未计人物 `grow`、属性、状态、专长和性格修正。

床的默认时长改为按当前精力动态计算，以恢复到 80 为目标；精力已达到 80 时仍可使用，
最短时长为 1.5 秒。

所有基础属性家具在属性已满时仍可使用。`stopWhenFull: true` 仅对“从未满开始，
本次恢复到允许上限”的行为提前结束；若开始时已经满值，不会立即取消动作。

性格化单次上限、少食拒绝和长期范围变化见
[`0615_基础需求范围与性格化家具.md`](0615_基础需求范围与性格化家具.md)。

## 9. 任务与离散恢复

任务系统支持 `need` 类型效果：

```js
{ type: 'need', target: 'assignee', key: 'hygiene', delta: 14 }
```

配置位置：

```js
questConfig.categoryCompletionNeedRewards
```

当前规则：

| 任务类别 | 完成奖励 |
| --- | --- |
| 洒扫 | 洁 `+14` |

所有“洒扫”类别任务完成时统一生效，包括个人差事、群体差事和日常任务。

互动模板也可通过 `effects` 离散增加或减少需求；这类效果不经过每日衰退公式，直接由
`CharacterEffectSystem` 加减。已有超额值不会因无关的离散效果被强制清回常态上限。

## 10. 状态持续时间与需求影响

状态系统位置：`js/systems/state.js`。

- `stateDefs[].duration` 单位是游戏分钟。
- `duration: -1` 表示永久或条件性消除。
- 永久状态运行时的 `remaining` 也统一为 `-1`。
- 旧配置的 `duration: 99999` 和旧存档中超长剩余时间会自动迁移为 `-1`。
- 状态每帧按 `现实秒 × 1.2` 扣减剩余游戏分钟。
- 性格可以通过 `stateDuration` 延长或缩短特定状态。
- 状态可通过 `needMods` 改变 `decay` 或 `grow`。

例：

```js
needMods: {
  energy: { decay: 1.5 },
  fun: { grow: 0.3 }
}
```

表示精力衰退倍率再乘 1.5，娱乐恢复倍率再乘 0.3。

`tempNeedMods[].remaining` 单位已统一为游戏分钟，与状态一致。版本 1/2 旧存档会在加载时将原现实秒剩余量换算为游戏分钟。

## 11. 关系自然衰退

配置：

```js
relationDecayPerDay: 1
```

关系每帧按游戏分钟结算：

```text
每日基础关系衰退 ÷ 1440 × 经过的游戏分钟 × 轴倍率
```

| 关系轴 | 每日倍率 | 当前正值每日变化 |
| --- | ---: | ---: |
| 好感 `affection` | 1.0 | -1.0 |
| 信任 `trust` | 0.3 | -0.3 |
| 友谊 `friendship` | 0.1 | -0.1 |
| 服从 `submission` | 0.05 | 缓慢回归身份基线 |

负好感和负信任以正值衰退速度的一半向 0 恢复。友谊只对正值衰退。

## 12. AI 与时间

- AI 快通道每游戏分钟执行一次。
- AI 慢通道每 15 游戏分钟执行一次，即约 12.5 现实秒。
- 时段变化时，所有人物 AI 候选缓存标记为需要重算。
- AI 在 22:00-05:00 强烈偏向睡眠。
- 05:00-09:00 偏向进食和梳洗。
- 18:00-21:00提高进食相关权重。
- 需求阈值、性格和家具可达性共同决定何时去恢复需求。

## 13. 任务时间单位

任务统一通过 `getGameTimestamp()` 使用游戏分钟时间戳：

```text
时间戳 = gameDay × 1440 + gameHour × 60 + gameMinute
```

| 模式 | 含义 |
| --- | --- |
| `GAME_HOURS` | 接受时间 + 参数小时 |
| `GAME_DAYS` | 接受时间 + 参数天数 |
| `BY_TIME_OF_DAY` | 当日指定小时；已错过则顺延一天 |
| `BY_PHASE` | 当前实现固定为接受后 180 游戏分钟 |
| `BEFORE_SLEEP` | 当日 22:00 |
| `NO_DEADLINE` | 无期限 |

常用配置：

- 待回应任务过期：180 游戏分钟，即 2.5 现实分钟。
- AI 尝试下发任务：每 60 游戏分钟，即 50 现实秒。
- 默认重复冷却：1440 游戏分钟，即 1 游戏日。

## 14. 存档与后台配置

存档键：`dgy_save`。

存档内容包括：

- `gameHour`、`gameMinute`、`gameDay`、`gamePeriod`
- 人物当前需求
- 人物长期需求范围习惯 `needMeta.adaptation`
- 状态及剩余时间
- 临时需求修正及剩余时间
- 关系值
- 任务时间、冷却与进度
- 家具占用和剩余使用时间
- AI 冷却

当前存档版本为 `3`。版本 1/2 可继续读取，并自动迁移临时需求修正与永久状态。

配置键：`dgy_config`。

加载时会先复制 `DEFAULT_CONFIG`，再用浏览器本地配置深度覆盖。因此：

- 只修改源码默认值，不一定会覆盖玩家后台已经保存的同名配置。
- 新增配置字段通常会从默认配置补入。
- 调参后验证异常时，应检查后台配置或清除 `dgy_config`。
- 旧存档会保留当时的需求值和状态剩余时间，但之后按新速率继续结算。

游戏日变化、任务变化、关系变化、互动完成和记忆变化会触发延迟 2.5 秒自动存档。

## 15. 调参操作顺序

### 改整局时间速度

修改 `js/runtime.js`：

```js
const GAME_DAY_REAL_MIN = 20;
```

增大该值会让一天更长，所有按游戏分钟运行的需求、状态、关系、任务和 AI，相对现实时间一起变慢。

### 改某项需求的全局节奏

优先修改 `js/config.js`：

```js
needDecayPerGameDay
```

不要先批量修改所有人物系数。全局值负责游戏节奏，人物系数负责人物差异。

### 改某个人物

修改：

```text
characters[].baseNeedCoeffs
```

调整唯一的 `decay` 字段。旧配置中的 `timeDecay` 会在加载时折入 `decay` 并删除。

### 改某种行为恢复量

修改家具模板：

```text
furnitureTemplates[].needRestores[].ratePerGameMin
furnitureTemplates[].duration
```

估算公式：

```text
总恢复 ≈ ratePerGameMin × duration × 1.2 × grow
```

### 改状态影响

修改：

```text
stateDefs[].needMods
stateDefs[].duration
```

### 改任务完成恢复

全类别统一效果修改：

```text
questConfig.categoryCompletionNeedRewards
```

单个任务效果修改：

```text
questConfig.templates[id].rewards
```

## 16. 推荐平衡标准

| 需求 | 推荐目标 |
| --- | --- |
| 饥 | 普通人物每 6-9 游戏小时需要一次正餐或点心 |
| 倦 | 普通人物清醒 16-24 游戏小时后明显需要睡眠 |
| 闷 | 普通人物每天进行 1-2 次娱乐活动 |
| 洁 | 普通人物每 2-4 游戏日进行一次完整梳洗，爱洁者更频繁 |

测试时至少观察：

1. 普通人物连续运行 1 游戏日的需求变化。
2. 体弱、嗜睡、贪嘴、爱洁、邋遢人物的差异。
3. 三餐行为是否足以抵消一天饥饿下降。
4. 一次睡眠是否足以支持下一段清醒时间。
5. AI 是否能及时找到对应家具。
6. 洒扫任务完成后洁净是否增加。
7. 本地 `dgy_config` 是否覆盖了新参数。

| 原问题 | 处理结果 |
| --- | --- |
| 家具恢复字段名与单位不一致 | 已统一为 `ratePerGameMin`，后台显示“恢复/游戏分”，旧字段自动迁移 |
| 临时需求修正使用现实秒 | 已统一为游戏分钟，旧存档自动换算 |
| 场景邀请监听不存在的 `time:minute` | 已改为监听 `time:tick` |
| 卡顿时补算受单帧 20 游戏分钟限制 | 已改为固定小步与 `simBacklog` 追赶；页面隐藏时明确暂停 |
| 家具恢复与自然衰退的执行关系不清 | 已明确为“正在恢复的同项需求暂停自然衰退，其他需求照常下降”，并集中到 `isNeedRestoringNow` 判断 |
| `decay × timeDecay` 双倍率易误调 | 已合并为唯一 `decay`；旧配置加载时自动折算 |
| `duration: 99999` 被当作永久状态 | 永久状态统一使用 `duration: -1` 和 `remaining: -1`，旧配置及旧存档自动迁移 |
| 满值需求导致家具无法使用或立即结束 | 已允许满值使用；仅从未满恢复到本次允许上限时提前结束 |
| 超过 100 的需求下一帧被夹回 | 已改为从超额值自然衰退 |
| 床固定时长与实际缺失精力无关 | 已按恢复到精力 80 动态计算，满 80 仍保留最短动作 |

### 仍需遵守的设计边界

- 页面隐藏期间不会推进游戏时间，这是明确的暂停策略，不是离线模拟。
- 每渲染帧最多追赶 1 现实秒模拟量；较大活动页卡顿会在后续数帧逐步追平，避免单帧长时间阻塞。
- 家具 `duration` 仍使用现实秒，因为它同时控制动作动画和玩家等待时间；恢复速率使用游戏分钟。

## 19. 角色行为遥测

为持续观察“一整天做了什么、为什么选择该行为”，新增
`js/systems/behavior-telemetry.js`。

### 默认策略

- 默认只观察林黛玉 `daiyu`，避免所有 NPC 的高频决策迅速占满浏览器存储。
- 数据保存在 `localStorage` 的 `dgy_behavior_telemetry_v1`。
- 循环保留最近 1200 条记录，每 20 条或跨游戏日时落盘。
- 普通慢通道每 15 游戏分钟记录一次；快通道只在实际替换行为时记录。
- 家具开始/完成/拒绝、需求范围变化、双向社交和任务下发/接受/完成/失败/拒绝另行记录。

### 每次决策记录

- 游戏日、时刻、AI 通道、当前状态和需求值/比例。
- 当前行动权重、替换阈值、是否真的替换。
- 原始候选数、保留候选数、按家具/社交/闲游分类的数量。
- 社交诊断：附近人数、冷却拦截数、规则拦截数、成功生成数。
- 权重前六名及各乘数：需求、性格、状态、时段、记忆、专长、距离、
  戏剧意图、人群、归巢性、任务。

### 调试接口

浏览器控制台可直接使用：

```js
BehaviorTelemetry.summarize('daiyu')
BehaviorTelemetry.query({ charId: 'daiyu', event: 'evaluation', limit: 20 })
BehaviorTelemetry.download('daiyu')
BehaviorTelemetry.watch('baoyu', true)
BehaviorTelemetry.watch('daiyu', false)
BehaviorTelemetry.getWatchedCharIds()
BehaviorTelemetry.clear('daiyu')
```

`download` 会下载可归档的 JSON。`watch` 设置单独保存在
`dgy_behavior_telemetry_settings_v1`，刷新页面后仍然有效。

## 20. 已知风险与后续整理项

### 社交候选可能根本没有进入抽选池

自主社交目前只为 `maxCandidateDistance = 25` 范围内的人生成候选。人物不在附近时，
不会主动把普通社交目标纳入候选；只有已经产生戏剧意图时，才可能生成跨距离
`seek` 行为。

此外，候选生成顺序是“家具 → 社交 → 闲游”，随后直接执行：

```js
pool.slice(0, cacheMaxSize)
```

`cacheMaxSize` 当前为 30。附近家具较多时，后加入的社交候选可能在计算权重前就被截掉。
遥测中的 `rawByKind.interaction`、`retainedByKind.interaction` 和
`poolTruncatedSocial` 用于确认此问题。

后续应将候选池改为分类配额或先算粗权重再截断，避免数组插入顺序决定 NPC 是否社交。

### 饱腹时食物仍保持明显正权重

`needUrgency` 在需求比例高于 80% 时仍返回 `0.1`，所以食物候选最低仍会获得：

```text
1 + 0.1 × hungerWeight(1.2) = 1.12
```

比例在 50%-80% 时返回 `0.8`，饥饿需求乘数会达到 `1.96`。黛玉初始饥饿为
`62/90 ≈ 69%`，因此开局已接近两倍进食权重。

黛玉归巢性较高，自宅的饭桌、点心案等还会再获得较强的
`homewardFactor`，所以即使没有饥饿危机，也容易反复抽中吃食。

后续应考虑：

- 饱腹区间让食物需求乘数回到 `1`，而不是继续加权。
- 正餐增加最短间隔或按早、中、晚餐时段加权。
- 点心设置更低基础权重，并与正餐共享进食冷却。

### 黛玉的琴台权重会发生多层叠乘

琴台标签为 `lundao + fun + instrument`。黛玉同时拥有：

- 旧性格 `shuchi`：`lundao ×2.0`
- 旧性格 `qinggao`：`lundao ×1.7`
- 新性格 `duochou`：`lundao ×1.3`
- 上午时段：`lundao ×1.4`
- 琴台位于自宅：继续获得归巢性加权

仅上述性格乘数理论上已可达到 `2.0 × 1.7 × 1.3 = 4.42`，尚未计入时段、
随机、距离、专长和归巢性，因此会明显压过普通家具和社交候选。

后续应为同标签的多性格叠乘设置上限，或改用边际递减合并；娱乐已接近满值时，
琴台等娱乐行为也应降权，而不是继续依赖 `stopWhenFull` 在动作阶段终止。

## 21. 基础需求：交游

配置键为 `social`，界面名称为“交游”。它表示角色近期是否与人来往，不等同于
关系值，也不等同于即时娱乐“闷”。

### 时间尺度

- 默认初始值：85。
- 每次有效社交完成后，重新获得 1440 游戏分钟（一整日）的不衰退期。
- 不衰退期结束后，每游戏日基础下降 34 点。
- 初始 85 点且完全不社交时，约在第 3.5 日见底。
- 刚恢复到 100 点后完全不社交，约在第 3.94 日见底。

人物、状态和性格仍可通过标准需求系数修改实际速度。

### 分段行为

| 数值 | 状态 | AI 行为 |
| --- | --- | --- |
| 80-100 | 满足 | 主动社交权重降为 0.45 倍，但仍会回应正常互动 |
| 40-80 | 平淡 | 社交与独处均按正常权重参与抽选 |
| 20-40 | 孤寂 | 社交权重约 3 倍，并生成跨场景“寻熟人说话”候选 |
| 5-20 | 寂寞难耐 | 社交权重约 6 倍，群聊加入率大幅提高，传情/调笑等冒进互动额外加权 |
| 0-5 | 形单影只 | 社交权重约 9 倍，并按角色触发特殊状态 |

0-5 的默认特殊状态：

- 宝玉：`chikuang`，犯痴狂。
- 黛玉：`ganshang`，临风洒泪。
- 社会等级 4 及以下的仆从：`socialPanic`，惶惶不安。
- 其他角色：`utterLoneliness`，形单影只。

### 恢复

双人互动完成后双方恢复交游，并重置一日缓冲：

| 类别 | 恢复 |
| --- | ---: |
| 寒暄/叙旧 | +24 |
| 论道 | +20 |
| 调笑 | +24 |
| 慰藉/倾诉 | +28 |
| 传情 | +30 |
| 争执 | +8 |
| 普通交谈 | +18 |
| 加入群聊 | +18 |

争执仍算“与人发生强烈接触”，因此少量恢复交游，但会显著损害心绪。

## 22. 基础需求：心绪

配置键为 `mood`，界面名称为“心绪”。它是长期心理稳定度；“闷”仍表示短期娱乐，
短暂玩乐不能完全替代长期心绪恢复。

### 时间尺度

- 默认初始值：75。
- 每游戏日自然下降 6 点。
- 不考虑任何事件时，从 75 到 0 约需 12.5 游戏日。
- 实际变化以事件冲击为主，重大打击可一次下降 35-50 点。

### 分段行为

| 数值 | 状态 | 系统效果 |
| --- | --- | --- |
| 80-100 | 安宁 | 书案/工作案效率 1.2 倍；LLM 社交语气温和从容 |
| 40-80 | 平常 | 正常效率与情绪反应 |
| 20-40 | 烦乱 | 工作效率 0.8 倍；负面状态更易触发；偏向慰藉或独处 |
| 5-20 | 郁结 | 工作效率 0.6 倍；慰藉、倾诉和独处行为显著加权 |
| 0-5 | 崩溃 | 工作效率 0.35 倍，并按性格触发崩溃状态 |

0-5 的默认表现：

- 黛玉、多愁、悲观：`despairPoem`，写绝命诗。
- 急躁、刻薄、耿直：`baonu`，大怒。
- 宝玉：`chikuang`，犯痴狂。
- 其他角色：`emotionalCollapse`，大哭或摔物。

### 事件冲击

代表性默认值：

| 事件 | 心绪变化 |
| --- | ---: |
| 恩断义绝状态 | -50 |
| 不孝/重大责罚 | -35 |
| 任务失败 | -18 |
| 感伤 | -18 |
| 恼怒 | -12 |
| 尴尬 | -10 |
| 负关系变化 | 按变化量约 0.8 倍扣除，单次最多 -20 |
| 任务完成 | +6 |
| 温馨 | +10 |
| 家宴 | +8 |
| 赏月赏花 | +6 |
| 家庭口角 | -15 |

### 恢复

- 慰藉类互动：双方基础 +12；“倾诉”的发起者另外 +10。
- 传情 +7，调笑 +5，普通叙旧/论道 +3。
- 书案 +4、琴台 +5、花圃 +5、凉亭 +4。
- 睡眠和普通休息只恢复 +2，不能用睡觉快速消除长期心结。

所有家具恢复均在动作完成时一次结算，不按每游戏分钟持续灌满。

## 23. AI 与候选池调整

交游低于 40 时，`CoreNeedSystem.extraCandidates` 会从可进入场景中选择关系较好的角色，
生成跨场景 `seek` 候选。抵达后，常规社交候选接管并选择具体互动。

候选池不再按“家具先加入”直接截取前 30 条，而是预留：

- 最多 10 条社交/寻人候选。
- 最多 16 条家具候选。
- 至少 2 条闲游或其他候选。

这样社交不会因为数组插入顺序在计算权重前被全部删除。

交游和心绪低于 10% 时不会进入“寻找生存家具”的紧急通道，避免把心理需求误判为
饥饿、睡眠或梳洗问题。
