## 1. 总目标

UI 是大观园模拟的操作层和解释层。它要让玩家能在同一屏内完成观察、选择人物、安排行动、查看人物、下发传令、阅读日志和保存进度。

设计目标：

- 第一屏就是可玩的模拟界面，不做落地页。
- 游戏画布保持主舞台，HUD 和面板只承载必要信息。
- 所有复杂系统都要有可解释入口：人物、起居、关系、任务、传令、技能、背包、存档。
- 玩家前台和策划后台信息口径一致，但展示方式不同。
- 古典题材视觉克制、清晰、可读，不让装饰压过信息。

## 3. 信息架构

```text
游戏主界面
  -> 顶部状态栏
  -> 中央 Canvas 模拟舞台
  -> 左侧任务 / 日志入口
  -> 行动队列
  -> 当前人物 HUD

  -> 通用弹窗 / 互动菜单 / LLM 浮层 / 后台
```

## 5. 交互原则

- 点击地面：移动或加入行动队列。
- 点击家具：打开家具互动菜单，选择动作。
- 点击人物：打开人物互动菜单，可社交、传令或查看人物。
- 点击当值“令”：打开传令面板并预选该人物。
- Shift 操作：用于追加或插队，具体由当前行动入口决定。
- 关系不足：不隐藏互动，显示意愿强弱和风险提示。
- 模型对白：必须可暂停、继续、终止；失败回退模板。
- 后台操作：保存状态要明确，批量导入要尽量可预检。

## 6. 视觉规范

| 项 | 口径 |
| --- | --- |
| 色彩 | 以纸色、青绿、金色、深墨为主；避免高饱和大色块。 |
| 字体 | 使用项目全局字体，不在局部页面随意改字体。 |
| 按钮 | 功能按钮使用清楚文字；图标按钮必须有 title。 |
| 面板 | 信息面板紧凑，卡片层级不要过深。 |
| 气泡 | 文案短，优先不遮挡人物和队列。 |
| 状态色 | 正面、负面、混合、风险要区分，但不过度鲜艳。 |
| 人物档案 | 简约古风，宣纸感、淡金线、玉色小签；性格用小标签。 |

## 7. 重点子系统

### 7.1 环形互动菜单

独立文档：`UI_环形互动菜单.md`。

维护重点：

- 菜单不能超出屏幕边缘。
- 中心区展示目标和返回/关闭。
- 第一层为类别，第二层为具体互动。
- 禁用项、风险项、LLM 项、意愿强弱要有视觉区分。

### 7.2 人物档案

独立文档：`持续更新_20_主界面与人物面板信息架构.md`。

维护重点：

- 前台人物档案是玩家可见总览：立绘、身份、家庭、职业、性格标签、六需求、健康/私库柱状图、声望小图。
- 后台人物设定是可编辑版本：同样信息口径，但允许修改配置。
- 前台不展示基础需求消耗/恢复系数，不展示后台 JSON。

### 7.3 后台 UI

独立文档：`持续更新_15后台与人物配置.md`。

维护重点：

- v2 后台走左侧分组 + 表格页 + 必要 JSON 高级入口。
- 策划不常用的场景配置/场景权限先隐藏。
- 表格编辑和 JSON 编辑保存口径必须一致。

### 7.4 LLM 对话 UI

维护重点：

- `waiting`、`generating`、`replying`、`paused`、`error` 状态要可感知。
- 换话题只显示当前情境可用互动。
- 对话浮层不能遮住关键菜单和错误信息。
