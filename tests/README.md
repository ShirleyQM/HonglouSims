# 大观园测试体系

> 目标：维护一套“策划可读、模块可扩展、发版前能跑”的测试脚本。  
> 原则：先保证核心链路不炸，再逐步把多日模拟和复杂耦合场景自动化。

## 1. 测试分层

### 1.1 配置体检

检查配置、源码入口、PRD 和关键系统是否缺失或引用断裂。

适合每次改配置后运行。

```bash
npm run test:config
```

当前覆盖：

- 核心文件是否存在。
- `index.html` 脚本顺序是否基本正确。
- 任务、传令、起居、关系、Utility AI 的关键代码标记是否存在。
- 关键 PRD 是否存在。
- 架构图是否包含传令、关系、起居、任务等核心依赖。

### 1.2 无头浏览器冒烟

检查页面能不能打开，核心 UI 能不能点，控制台有没有致命 JS 错误。

运行前需要先启动本地服务：

```bash
npm run start:local
npm run test:smoke
```

当前覆盖：

- 首页加载。
- 无 `pageerror`。
- 画布存在。
- 底栏关键入口存在。
- 传令入口存在。

### 1.3 后台冒烟

检查后台 v2 能不能打开，左侧所有策划入口能不能逐个进入。

```bash
npm run test:admin
```

当前覆盖：

- 后台 overlay 打开。
- v2 左侧导航存在。
- 每个 v2 导航项点击后不抛 JS 错误。
- 策划分组存在：人物配置、关系与身份、任务与传令、行为规则、世界配置、AI 与调试。

### 1.4 传令冒烟

检查新传令面板能不能打开，并显示点名/群体两栏。

```bash
npm run test:command
```

当前覆盖：

- 底栏“传令”打开统一面板。
- 面板包含“点名传令”和“群体传令”。
- 点名传令能列出可传令对象。
- 当值“令”角标能打开面板并预选人物。
- 候选卡展示服从/信任/预计接受等关系信号。

### 1.5 场景测试

检查跨系统链路是否符合游戏设计。

```bash
npm run test:scenario
```

第一版先维护场景注册表和报告，不强行伪造深度模拟。后续逐个自动化。

## 2. 发版前推荐命令

```bash
npm run test:release
```

这会按顺序执行：

- `test:config`
- `test:smoke`
- `test:admin`
- `test:command`
- `test:scenario`

如果只是快速检查配置：

```bash
npm run test:config
```

如果准备给玩家试玩：

```bash
npm run start:local
npm run test:release
```

## 4. 模块测试矩阵

| 模块 | 必测内容 |
| --- | --- |
| 人物 | 初始位置、需求、状态、性格、职业、家庭是否正常。 |
| 需求 | 衰退、危机、家具恢复、状态变化。 |
| 起居 | 不重叠、跨日、覆盖兜底、影响 Utility AI。 |
| 家具 | 可达、可用、恢复需求、技能限制、旁观反应。 |
| 社交 | 可用条件、关系变化、状态影响、LLM 兜底。 |
| 关系 | 友谊、信任、姻缘、服从方向性。 |
| 传令 | 点名、群体、权限、服从影响、任务生成。 |
| 任务 | 接受、拒绝、执行、完成、失败、质量。 |
| 随侍 | 当值、跟随、需求打断、归队。 |
| 场景 | 地图边界、权限、路径可达、空白地图保护。 |
| 后台 | v2 导航、表格打开、保存不崩。 |
| AI | 候选池、权重、作息/任务/需求优先级。 |

## 5. 核心耦合链路

不要只测试单个函数。这个游戏最容易坏的是系统链路：

```text
起居
  -> Utility AI 权重
  -> 家具 / 社交 / 任务候选
  -> 行动队列
  -> 需求 / 关系 / 状态变化
  -> 下一轮 AI 决策
```

以及：

```text
传令
  -> 身份 / 职业 / 仆从契约 / 关系服从
  -> 任务下发
  -> 接受概率
  -> 执行质量
  -> 忠诚 / 积怨 / 服从 / 信任反馈
```

## 6. 场景测试清单

### 袭人早上随侍宝玉

前置：

- 当前时间 07:00。
- 宝玉今日随侍是袭人。
- 袭人服从较高。
- 袭人没有严重需求危机。

期望：

- 传令面板显示袭人。
- 袭人候选卡显示服从和预计接受。
- 起居里有随侍投影。
- Utility AI 倾向跟随或靠近宝玉。
- 若饥饿、洁净、精力过低，会请辞解决需求后归队。

### 黛玉午后读书

前置：

- 当前时间在职业/技能活动附近。
- 黛玉可使用书桌或读书相关家具。
- 没有高优先级危机。

期望：

- 起居技能/职业块影响 Utility AI。
- 书房/读书相关家具候选获得加权。
- 行为后技能或需求状态有可解释变化。

### 群体传令洒扫

前置：

- 发布者有群体传令权限。
- 存在符合条件的仆从。

期望：

- 传令面板右侧显示群体传令任务。
- 展示命中人数。
- 下发后生成批次任务。
- 不符合身份或职责的人不被命中。

### 需求危机打断任务

前置：

- 某人已有随侍或普通任务。
- 饥饿/洁净/精力进入危机。

期望：

- 严重需求危机优先级高于普通起居和普通任务。
- 解决需求后，如果任务仍有效，会回归。

### 地图边缘保护

前置：

- 多人社交、跟随或寻路发生在场景边缘。

期望：

- 人物不会被推到空白地图。
- 目标点不可达时有兜底位置。
- 不会出现长时间卡死。

## 7. 报告规范

每个脚本会输出：

- 终端摘要。
- `tests/reports/latest-*.json`。
- `tests/reports/latest-*.md`。

报告分三级：

- `PASS`：必须满足。
- `WARN`：需要关注，但不阻断发版。
- `FAIL`：阻断发版。

## 8. 新增测试脚本规范

新增脚本时遵守：

- 文件放在对应目录。
- 使用 `tests/lib/test-harness.mjs`。
- 输出报告到 `tests/reports/`。
- 命名用 `validate-*`、`smoke-*`、`scenario-*`。
- 不直接修改游戏配置。
- 不依赖本地存档。

## 9. 当前建议

第一阶段重点让“上线试玩”安全：

- 保证页面和后台能打开。
- 保证传令、起居、任务、关系、家具这些入口不崩。
- 保证 PRD 和架构图能追踪到主要系统。

第二阶段再把多日模拟做深：

- 固定随机种子。
- 固定起始时间和人物位置。
- 记录人物、时辰、起居项目、是否遵循、六需求水平。
- 输出 Markdown 回放报告。
